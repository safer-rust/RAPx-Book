# Chapter 8.4. The Verification Pipeline

This chapter walks through the verification pipeline using the [`linked_list_nonnull`](https://github.com/safer-rust/RAPx/blob/main/rapx/tests/verify_cases/linked_list_nonnull/src/lib.rs) case study ‚Äî a doubly-linked list built with `NonNull<Node>` pointers:

```rust
#[rapx::invariant(Align(head.unwrap_some(), Node))]
#[rapx::invariant(Allocated(head.unwrap_some(), Node, 1))]
#[rapx::invariant(Owning(head.unwrap_some()))]
struct LinkedList { head: Option<NonNull<Node>>, tail: Option<NonNull<Node>> }
```

For each target the pipeline runs: **path extraction ‚Üí backward slicing ‚Üí forward verification ‚Üí SMT check**. The sections below use `from_vec` as the example.

## 8.4.1 Path Extraction

The first stage enumerates acyclic paths from function entry to each unsafe callsite using `PathGraph`'s SCC decomposition (see [¬ß5.1 Path Analysis](./5.1-path.md) for the full algorithm). Each path is a sequence of basic block IDs.

The `for value in values { ‚Ä¶ }` loop contains an SCC with two branches (`tail` is `None` or `Some`). The path extractor unrolls this SCC once, producing paths for the first iteration (both branches) and subsequent iterations (each branch repeated). The sections below use the **first Ptr2Ref path** as the single running example:

```
path [0, 1, 2, 3, 5, 7, 8, 9, 11, 14, 2, 3, 5, 7, 8, 9, 10]   ‚Üí   Ptr2Ref | Proved
```

This path represents a two-element input: the first iteration (empty list, bb11 initialises head and tail), then the second iteration where `list.tail` is `Some` and `tail.as_mut()` is called at bb10.

Paths are stored in a `PathTree` per callsite and capped at 1024. By default, verification starts at `repeat = 0` ‚Äî each SCC body at most once. After the first round, **auto-detectors** (see [¬ß8.4.5](#845-auto-repeat-detectors)) decide whether deeper unrolling is needed.

## 8.4.2 Backward Slicing

Take the path `[0, 1, 2, 3, 5, 7, 8, 9, 11, 14, 2, 3, 5, 7, 8, 9, 10]` ‚Äî two-element input, first iteration initialises the empty list, second iteration calls `tail.as_mut()` at bb10. The relevant MIR blocks:

```
bb0:  {
    _2 = LinkedList { head: None, tail: None }   // empty list
}
bb1:  {
    _8 = <Vec<i32> as IntoIterator>::into_iter(move _1) ‚Üí bb2
}
bb2:  {
    _11 = <IntoIter<i32> as Iterator>::next(&mut _8) ‚Üí bb3   // for loop body
}
bb3:  {
    switchInt(discriminant(_11)) ‚Üí [0: bb6, 1: bb5]   // None ‚Üí exit, Some ‚Üí process
}
bb5:  {
    _17 = Box::<Node>::new(move _18) ‚Üí bb7             // heap allocate
}
bb7:  {
    _24 = move _17
    _23 = Box::<Node>::leak(move _24) ‚Üí bb8            // transfer ownership
}
bb8:  {
    _22 = NonNull::<Node>::from(move _23) ‚Üí bb9        // wrap in NonNull
}
bb9:  {
    _25 = discriminant((_2.1: Option<NonNull<Node>>))
    switchInt(move _25) ‚Üí [0: bb11, 1: bb10]           // tail is None? Some?
}
bb10: {
    _30 = copy ((_2.1 as Some).0: NonNull<Node>)       // extract stored NonNull
    _34 = &mut _30
    _33 = NonNull::<Node>::as_mut(move _34) ‚Üí bb12     // ‚Üê CHECKPOINT
}
bb11: {
    (_2.0) = Some(_27)                                  // set head
    (_2.1) = Some(_29)  // _29 = copy _22              // set tail
}
bb14: {
    goto ‚Üí bb2                                          // loop back
}
```

The example path reaches bb10 **on the second iteration**: the first iteration went through bb11 (tail was `None`, initialised), then bb14 loops back, bb2-5-7-8 allocates a second node, bb9 dispatches again ‚Äî this time tail is `Some`, so control flows to bb10.

The `BackwardSlicer` extracts only the MIR statements that contribute to the target place's value ‚Äî everything else is dropped.

Walk backward from `_33 = as_mut(move _34)`:

| Step | Collected | Provides |
|------|-----------|----------|
| bb10 | `_34 = &mut _30` | `_34` ‚Üí `_30` |
| bb10 | `_30 = ((_2.1 as Some).0)` | `_30` ‚Üí `list.tail` |
| bb11 | `(_2.1) = Some(_22)` | `list.tail` ‚Üí `_22` |
| bb8 | `_22 = NonNull::from(_23)` | `_22` ‚Üí `_23` |
| bb7 | `_23 = Box::leak(_17)` | `_23` ‚Üí `_17` |
| bb5 | `_17 = Box::new(_18)` | allocation origin ‚Üí **stop** |

The collected chain is `_34 ‚Üí _30 ‚Üí tail ‚Üí _22 ‚Üí _23 ‚Üí _17`. The slicer recognises standard-library calls by their effects: `Box::new` ‚Üí `BoxAllocation`, `NonNull::from` ‚Üí `ReturnPointerFromArg`.

No `ContractFact` items are injected ‚Äî `from_vec` has no `#[rapx::requires]` of its own.

## 8.4.3 Forward Verification

The `ForwardVerifier` performs abstract interpretation on each MIR statement along the path, accumulating facts that feed into the `Ptr2Ref` check at the checkpoint. `Ptr2Ref` is defined as `Init(p, T, 1) && Align(p, T) && Alias(p, ret)` per primitive-sp; the current SMT implementation checks `NonNull` and `Align` as a simplified subset, while `Alias` is handled in the hazard tracking phase below.

### 8.4.3.1 Fact accumulation before the checkpoint

| BB | MIR statement | Derived fact |
|----|--------------|--------------|
| bb5 | `_17 = Box::<Node>::new(move _18)` | `KnownNonZero(_17)` <br> `KnownAllocated(_17, Box<Node>, 1)` <br> `KnownInit(_17.0.0, Node, 1)` |
| bb7 | `_23 = Box::<Node>::leak(move _24)` | (tracks _23 ‚Üí _17) |
| bb8 | `_22 = NonNull::<Node>::from(move _23)` | `PointsTo(_22, _17)` |
| bb11 | `(_2.1) = Some(_22)` | `FieldAssign(tail, _22)` |
| bb14 | `goto ‚Üí bb2` (loop back-edge) | path-locals reset; `FieldAssign` and `KnownAllocated` persist |
| bb10 | `_30 = copy ((_2.1 as Some).0)` | `PointsTo(_30, _17)` (resolve tail ‚Üí _22 ‚Üí _17) |
| bb10 | `_34 = &mut _30` | `PointsTo(_34, _17)` |
| bb10 | `_33 = NonNull::<Node>::as_mut(move _34)` | resolve `_34 ‚Üí _17`; Ptr2Ref discharged |

At the checkpoint, the verifier resolves the value chain `_34 ‚Üí _30 ‚Üí tail ‚Üí _22 ‚Üí _23 ‚Üí _17`. Three facts are available from `Box::new`:

| Fact | Purpose |
|------|---------|
| `KnownNonZero(_17)` | `_17 ‚âÝ 0` ‚Üí proves `NonNull` component of Ptr2Ref |
| `KnownInit(_17.0.0, Node, 1)` | inner memory of Box is initialized with a valid `Node` ‚Üí would prove `Init` |
| alignment from allocator | `_17` is aligned for `Box<Node>` ‚Üí proves `Align` component |

The current Ptr2Ref SMT check uses `NonNull` + `Align`; `Init` is not yet lowered to SMT (tracked separately via the Typed/Init property checkers). Both discharged ‚Üí `Ptr2Ref | Proved`. The remaining `Alias` component is handled below.

### 8.4.3.2 Hazard tracking after the checkpoint

`as_mut` also declares the `Alias(p, ret)` hazard: the returned `&mut` reference `_33` aliases the original pointer `_17`, violating Rust's exclusive mutability principle. Proving Ptr2Ref alone is insufficient ‚Äî the verifier must also check that this hazard does not cause UB in the remainder of the function, and that it is resolved before the function exits.

The verifier performs a **flow-sensitive forward scan** from the checkpoint:

1. **Scope analysis** ‚Äî `destination_flows_to_return?` checks whether `_33` escapes the function (assigned to `_0` or stored in a struct field that is returned). In this example, `_33` is used only to set `prev`/`next` pointers in bb12-13, then goes out of scope when the loop ends ‚Äî **not returned**.

2. **Local conflict scan** ‚Äî since the hazard is local, the verifier scans all remaining MIR statements on the path for conflicting accesses through the original pointer (`_17`, `_30`, `_34`). None are found.

3. **Resolution** ‚Äî `_33` dies at the end of its scope (loop body). At function exit, no hazardous alias remains.

‚Üí `Alias` hazard discharged. The function is `SOUND`.

**Contrast:** if `_33` were returned to the caller (e.g., `fn get_tail_mut(&mut self) -> &mut Node { unsafe { self.tail.as_mut() } }`), step¬Ý1 would detect the escape, and the Alias hazard would transfer to the caller ‚Äî the current function would still be SOUND with a hazard warning, while the caller inherits the obligation to ensure the aliased pointer is not concurrently used.

## 8.4.4 SMT Check

The forward facts are lowered to SMT assertions. For `Ptr2Ref`, the verifier resolves `_34 ‚Üí _17`:

```
KnownNonZero(_17)  ‚Üí  _17 ‚âÝ 0
```

The SMT solver uses this fact together with the allocator's alignment guarantee to discharge Ptr2Ref ‚Üí `Proved`.

The struct invariant check at bb15 follows the same chain ‚Äî `head.unwrap_some()` ‚Üí `_17` ‚Äî and `KnownAllocated(_17, Box<Node>, 1)` from `Box::new` discharges `Allocated`, while the `Align(head.unwrap_some(), Node)` and `Owning(head.unwrap_some())` invariants are proven by the same allocation facts. For `drop`, the `discharge_from_field_invariant` mechanism resolves callee obligations directly from the struct's invariant annotations.

## 8.4.5 Auto-Repeat Detectors

Loop-carried properties (bounds checks, heap allocations) sometimes need more than one loop iteration to verify. The verifier starts at `postfix_repeat = 0` (each SCC body appears at most once). After the first round, dedicated auto-detectors run to decide whether deeper unrolling would help.

### Phase 1.5 ‚Äî InBound

The InBound detector addresses **off-by-one errors**. At repeat 0, a bounds check inside a loop may appear `Proved` because the path only exercises the first iteration. A bug that triggers on the second iteration (e.g. `index == len`) would be missed.

- **Trigger**: all InBound results are `Proved` at the current depth.
- **Action**: progressively unroll up to 16 levels. At each level, re-run verification.
- **Stop conditions**: an InBound check becomes `Unproved` (found a bug ‚Äî report it); or the `Proved` set stabilises (same as a previous level); or the `Proved` set oscillates between two states.

### Phase 1.6 ‚Äî Deref / ValidPtr

Same logic as InBound, applied to `Deref` and `ValidPtr` results. These are compound properties (`Allocated ‚àß InBound`) that depend on loop-carried allocation and bounds facts.

### Phase 1.7 ‚Äî Allocated / Deref / ValidPtr

This detector handles **allocation reachability**. Fresh allocations inside loops (e.g. `Box::new` in a `for` body) produce facts (`PointsTo`, `KnownAllocated`) that the slicer needs to reach. The problem: at repeat 0 the allocation site appears on the path, but the fact may be too shallow in the trace for the forward verifier to carry it all the way to the callsite. Deeper unrolling repeats the loop body, extending the trace so the slicer can bridge the gap.

- **Trigger**: any Allocated / Deref / ValidPtr result is `Unproved` (excluding precision loss / connectivity failures).
- **Action**: retry with progressively deeper unrolling (up to 16). A new `VerifyDriver` is spawned at each level.
- **Stop conditions**: all targeted properties become `Proved`; or the set of still-unproved results converges to a previous level; or it oscillates.

In all three phases, earlier results from lower repeat levels are retained. Only newly `Proved` results from deeper levels are merged in ‚Äî the mechanism is monotonic and never weakens a verdict.
