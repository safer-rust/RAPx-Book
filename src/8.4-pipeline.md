# Chapter 8.4. The Verification Pipeline

The core verification flow is implemented across three stages orchestrated by `VerifyEngine` ([`engine.rs`](https://github.com/safer-rust/RAPx/blob/main/rapx/src/verify/engine.rs)) and driven by `VerifyDriver` ([`driver.rs`](https://github.com/safer-rust/RAPx/blob/main/rapx/src/verify/driver.rs)):

```
VerifyDriver::verify_function()
  Ōöé
  Ōö£ŌöĆ 1. Path Extraction (path.rs / PathGraph)
  Ōöé     Build finite, acyclic paths from CFG to each callsite
  Ōöé     using SCC-aware enumeration (PathGraph ŌĆö [Chapter 5.1](./5.1-path.md)).
  Ōöé
  ŌööŌöĆ 2. Callsite Verification (engine.rs)
        Ōö£ŌöĆ 2a. Backward Visit ŌĆö collect MIR items relevant to the property
        Ōö£ŌöĆ 2b. Forward Visit  ŌĆö build abstract state (values, facts)
        ŌööŌöĆ 2c. SMT Check      ŌĆö encode facts + property, invoke Z3
```

This chapter also covers the two structure-oriented analyses that build on the same pipeline ŌĆö struct invariant verification ([┬¦8.4.6](#846-struct-invariant-verification)) and invless-mode sequences ([┬¦8.4.7](#847-invless-mode-sequences)) ŌĆö and the raw-pointer dereference checking ([┬¦8.4.8](#848-raw-pointer-dereference-checking)).

## 8.4.1 Path Extraction

[`PathExtractor`](https://github.com/safer-rust/RAPx/blob/main/rapx/src/verify/path_extractor.rs) ([`path.rs`](https://github.com/safer-rust/RAPx/blob/main/rapx/src/verify/path.rs)) builds finite verification paths from the function's CFG to each unsafe callsite. It uses `PathGraph` (from [`analysis::path_analysis::graph`](https://github.com/safer-rust/RAPx/blob/main/rapx/src/analysis/path_analysis/graph.rs)) to:

1. Build the CFG and detect strongly connected components (SCCs).
2. Enumerate acyclic paths through the CFG, allowing controlled SCC postfix repetition via `postfix_repeat`.
3. Filter paths that reach the target callsite block.
4. Truncate at the target block and deduplicate by prefix, storing them in a `PathTree` per callsite.

Each `Path` is a sequence of `PathStep::Block` steps ending with a `PathStep::Callsite`. Path enumeration is capped at `PATH_LIMIT` (1024) per callsite.

### Why Path Sensitivity Matters

Consider a function with two branches that lead to the same unsafe call, but one branch establishes the needed preconditions and the other does not:

```rust
unsafe fn example(cond: bool, ptr: *mut u32, len: usize) {
    if cond {
        assert!(len > 0);
        let _ = ptr::read(ptr);  // Safe: len > 0 guarantees at least one element
    } else {
        let _ = ptr::read(ptr);  // Unsafe: len may be 0
    }
}
```

A path-insensitive analysis would merge the states from both branches, losing the `len > 0` fact and incorrectly flagging both call sites. Path sensitivity preserves each branch's context, allowing the verifier to prove the first `ptr::read` while correctly identifying the second as unproved.

### SCC Decomposition and Multi-Occurrence Checkpoints

SCC decomposition handles loops by identifying the dominator (enter block) of each SCC and decomposing the SCC tree hierarchically. Path reachability filtering ([Chapter 5.1](./5.1-path.md)) further prunes paths that are structurally possible but semantically impossible.

When a checkpoint lies inside a loop body (an SCC), the path tree contains **multiple occurrences** of the same basic block at different iteration depths. The backward slicer's `build_leaf_items` processes the tree in post-order and continues past the checkpoint into its children (**child-path expansion**), producing one leaf per iteration count. This enables checking safety properties at multiple iteration depths ŌĆö critical for off-by-one errors that only manifest after many iterations.

### Postfix Repeat Convergence

When `--postfix-repeat` is not explicitly set, the verifier automatically expands the postfix repeat value, exploring progressively deeper loop unrollings until convergence:

| `postfix_repeat` | Max loop iterations in path |
|-----------------|---------------------------|
| 0 | 1 |
| 1 | 2 |
| k | k + 1 |

Auto-expansion stops when a property becomes `Unproved`/`Unknown`, when the proven set stabilizes, or when an oscillation between two states is detected. The following example only fails at depth because an early-return guard anchors a minimum length:

```rust
#[rapx::verify]
pub fn unsound_len_guard_off_by_one(data: &[u32]) {
    let len = data.len();
    if len < 10 { return; }        // anchors len >= 10
    let ptr = data.as_ptr();
    let mut i = 0usize;
    while i < len {
        let current = ptr.wrapping_add(i + 1);
        unsafe { require_scc_inbound(current); }
        i += 1;
    }
}
```

At `repeat=9` (10 body executions) the InBound check for `ptr.wrapping_add(10)` requires `10 < len` (i.e. `len >= 11`), but only `len >= 10` is known ŌåÆ **Unproved ŌåÆ UNSOUND**.

## 8.4.2 Backward Visit

The `BackwardVisitor` ([`path_refine/visitor.rs`](https://github.com/safer-rust/RAPx/blob/main/rapx/src/verify/path_refine/visitor.rs)) starts from the callsite and walks backward along the path, collecting only MIR statements and terminators that are *relevant* to the property being checked. Items are classified as:

- **`Statement`**: A MIR statement whose def-use chain reaches the property's target place.
- **`Terminator`**: A MIR terminator (call, switch, assert) that influences control-flow constraints.
- **`ContractFact`**: A caller's `#[rapx::requires]` contract injected as an entry assumption.
- **`Forget`**: A conservatively dropped fact (e.g., after an unknown call that may mutate a tracked pointer).

The backward visitor uses def-use analysis ([`def_use.rs`](https://github.com/safer-rust/RAPx/blob/main/rapx/src/verify/def_use.rs)) to trace value flow from the callsite arguments back to their origins, filtering out irrelevant operations. It also consults `CallDependencySummary` to determine which callee arguments the return value depends on.

## 8.4.3 Forward Visit

The `ForwardVisitor` ([`forward_visit.rs`](https://github.com/safer-rust/RAPx/blob/main/rapx/src/verify/forward_visit.rs)) walks forward along the retained MIR items, building an abstract state consisting of:

- **`values`**: A map from MIR locals to `AbstractValue` (Place, Ref, RawPtr, ConstInt, Cast, Binary op, CallResult).
- **`facts`**: A list of `StateFact` entries: `PointsTo`, `KnownAligned`, `KnownInit`, `KnownAllocated`, `KnownNonZero`, `BranchEq`, `Call/CallEffect`.

The forward visitor processes each path independently, so facts established on one branch do not leak into another.

## 8.4.4 SMT Check

The `SmtChecker` ([`smt_check/`](https://github.com/safer-rust/RAPx/tree/main/rapx/src/verify/smt_check/)) encodes forward state facts and the target property as SMT-LIB assertions and invokes Z3 to check whether the facts logically imply the property. Each property kind has a dedicated checking module:

| Module | Checks |
|--------|--------|
| `valid_ptr.rs` | Pointer validity for the given type and count |
| `align.rs` | Pointer alignment matches the type requirement |
| `in_bound.rs` | Pointer arithmetic within allocation bounds; slice index bounds |
| `init.rs` | Memory initialization state |
| `non_null.rs` | Pointer non-nullness |
| `allocated.rs` | Memory allocation tracking |
| `valid_num.rs` | Numeric range constraints |
| `deref.rs` | Composite: `Allocated` + `InBound` for dereference safety |
| `non_overlap.rs` | Memory region non-overlap constraints |
| `typed.rs` | Memory holds a valid bit-pattern for the type |
| `alias.rs` | No other live pointer aliases the memory |
| `alive.rs` | Allocation liveness tracking |

Properties without dedicated SMT modules produce `CheckResult::Unknown` and do not contribute to soundness verdicts. RAPx uses the `z3` crate for native bindings; the solver runs under a timeout (default 5 seconds per check), and a timeout yields `CheckResult::Unproved`.

## 8.4.5 Call Effect Summaries

[`call_summary.rs`](https://github.com/safer-rust/RAPx/blob/main/rapx/src/verify/call_summary.rs) maintains a database of call effects for common unsafe and standard library functions. Key effects:

| Effect | Description | Examples |
|--------|-------------|----------|
| `ReturnAliasArg` | Return aliases an argument | `NonNull::as_ptr` |
| `ReturnPointerAdd` | Return = base + offset ├Ś stride | `ptr::add`, `ptr::wrapping_add` |
| `ReturnPointerSub` | Return = base ŌłÆ offset ├Ś stride | `ptr::sub` |
| `ReturnNonZero` | Return is guaranteed non-zero | `NonNull::new_unchecked` |
| `ReturnAligned` | Return has known alignment | `alloc::alloc` |
| `ReturnConst` | Return is a known constant | `size_of`, `align_of` |
| `ReturnLengthOfArg` | Return = len(arg) | `slice::len` |
| `ReadMemory` | Call reads from pointed-to region | `ptr::read` (src) |
| `WriteMemory` | Call writes (establishes `KnownInit`) | `ptr::write` (dst) |
| `ForgetArgFacts` | Call invalidates facts about an arg | `ptr::swap` |

These summaries bridge intra-procedural path analysis and the semantics of standard library functions, avoiding full inter-procedural MIR analysis.

## 8.4.6 Struct Invariant Verification

When a function target is a method on a struct with `#[rapx::invariant(...)]` annotations, the verifier additionally checks that the struct invariants are maintained:

- **Constructors** (functions returning `Self`): Paths are filtered to return blocks. Invariants must be proved at each return checkpoint, with only the constructor's own `#[rapx::requires]` as entry assumptions.
- **Methods**: Invariants are assumed at function entry (prepended as `ContractFact` items) and verified at all return points.

The `verify_struct_invariants` method in `driver.rs` handles this via `build_invariant_paths()`.

### Worked Example: Bounded Buffer

```rust
#[rapx::invariant(ValidPtr(self.ptr, u8, self.cap))]
#[rapx::invariant(Align(self.ptr, u8))]
pub struct Buf {
    ptr: *mut u8,
    len: usize,
    cap: usize,
}

impl Buf {
    #[rapx::requires(ValidPtr(ptr, u8, cap))]
    #[rapx::requires(Align(ptr, u8))]
    pub unsafe fn new(ptr: *mut u8, cap: usize) -> Self {
        Buf { ptr, len: 0, cap }
    }

    pub unsafe fn push(&mut self, val: u8) {
        // Verifier assumes on entry: ValidPtr(self.ptr, u8, self.cap), Align(self.ptr, u8)
        assert!(self.len < self.cap);
        self.ptr.add(self.len).write(val);
        self.len += 1;
        // Verifier re-checks invariants on exit.
    }
}
```

For `push`: (1) invariants are assumed at entry; (2) the `assert!(self.len < self.cap)` provides the `ValidNum` constraint that proves `ValidPtr`/`Align` for the offset pointer; (3) at exit `self.ptr` and `self.cap` are unchanged, so both invariants still hold. Proved.

## 8.4.7 Invless Mode Sequences

In `invless` mode (`run_invless_sequences` in `driver.rs`), when a struct has constructors and methods, the verifier generates verification sequences:

```
Constructor ŌåÆ Method
Constructor ŌåÆ Mutator ŌåÆ Method
Constructor ŌåÆ MutatorŌéü ŌåÆ MutatorŌéé ŌåÆ Method
```

Each sequence propagates the constructor's `#[rapx::requires]` contracts through the mutator chain. Mutated fields are tracked via `get_mutated_fields`, and contracts referencing those fields are filtered out at each step. For each sequence, the verifier builds a `FunctionTarget` with the accumulated `caller_requires`, then runs the standard pipeline.

### Sequence Generation Algorithm

```
For each struct S:
  For each constructor C of S (returns Self):
    contracts = C's #[rapx::requires]
    For each &self method R of S:                       // Constructor ŌåÆ Reader
      Verify R with `contracts` as caller_requires
    For each &mut self method M of S:                   // Constructor ŌåÆ Mutator ŌåÆ ŌĆ”
      mutated = get_mutated_fields(M's body)
      next_contracts = drop contracts referencing mutated fields
      For each &self method R of S:
        Verify R with `next_contracts` as caller_requires
      ... (continue chaining up to MAX_CHAIN_DEPTH)
```

`invless` mode is best used as a scoping tool ŌĆö it tells you *which* constructor contracts are implicitly assumed by each method sequence. An unproved sequence indicates either that the constructor's contracts are insufficient for the reader, or that a mutator invalidates a contract the reader depends on. This helps identify where explicit `#[rapx::invariant]` annotations are needed.

## 8.4.8 Raw Pointer Dereference Checking

In addition to explicit unsafe call sites, the verifier checks raw pointer dereferences (`*ptr` and `*ptr = val`) in unsafe blocks. `build_raw_ptr_deref_checks()` ([`target.rs`](https://github.com/safer-rust/RAPx/blob/main/rapx/src/verify/target.rs)) scans the MIR for dereference projections on raw pointer types and creates pseudo-callsites:

- **Read dereferences** (`*ptr`): Requires `ValidPtr(ptr, T, 1)`, `Align(ptr, T)`, and `Typed(ptr, T)`.
- **Write dereferences** (`*ptr = val`): Requires `ValidPtr(ptr, T, 1)` and `Align(ptr, T)`.

These pseudo-callsites use the same pipeline, so they benefit from the same path sensitivity and abstract state tracking as regular unsafe calls.

```rust
unsafe fn write_first_element(slice: &mut [u32]) {
    let ptr = slice.as_mut_ptr();
    *ptr = 42;   // write-dereference pseudo-callsite: ValidPtr(ptr, u32, 1), Align(ptr, u32)
}
```

The verifier traces `ptr` back to `slice.as_mut_ptr()`, whose call effect summary establishes that `ptr` points into `slice`'s allocation and is properly aligned. Both properties are Proved.
