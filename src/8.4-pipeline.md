# Chapter 8.4. The Verification Pipeline

This chapter walks through the verification pipeline using the [`linked_list_nonnull`](https://github.com/safer-rust/RAPx/blob/main/rapx/tests/verify_cases/linked_list_nonnull/src/lib.rs) case study — a doubly-linked list built with `NonNull<Node>` pointers. The struct declares three invariants:

```rust
#[rapx::invariant(Align(head.unwrap_some(), Node))]
#[rapx::invariant(Allocated(head.unwrap_some(), Node, 1))]
#[rapx::invariant(Owning(head.unwrap_some()))]
struct LinkedList { head: Option<NonNull<Node>>, tail: Option<NonNull<Node>> }
```

We'll use this report to explain each stage of the pipeline.

## 8.4.1 Path Extraction

The first stage enumerates all acyclic paths from function entry to each unsafe callsite. For `from_vec`, the loop creates multiple paths:

- `path [0, 1, 2, 3, 5, 7, 8, 9, 11, 14, 2, 3, 5, 7, 8, 9, 10]` — loop body, `list.head = Some(node)` branch
- `path [0, 1, 2, 3, 5, 7, 8, 9, 11, 14, 2, 3, 5, 7, 8, 9, 10, 12]` — loop body, `list.tail` branch (appends to existing tail)

For `drop`, the loop generates:

- `path [0, 1, 2]` — loop exits immediately (`head` is `None`)
- `path [0, 1, 2, 3, 4]` — one iteration: `as_ref` → `from_raw`
- `path [0, 1, 2, 3, 4, 5, 6, 1, 2, 3, 4]` — two iterations (truncated in output for brevity)

Path extraction uses `PathGraph`'s SCC decomposition to handle loops: each SCC's dominator is identified, and postfix repetition controls how many times the loop body is traversed. Paths are stored in a `PathTree` per callsite and capped at 1024.

When `--postfix-repeat` is not explicitly set, the verifier auto-expands until convergence. Phase 1.5 (InBound) tries up to 16 extra unrolls; phase 1.6 (Align) and 1.7 (Allocated/ValidPtr/Deref) build on prior results.

## 8.4.2 Backward Slicing

For each path, the `BackwardSlicer` walks backward from the callsite, collecting only MIR items relevant to the property being checked.

For the `Ptr2Ref` check at `NonNull::as_mut` in `from_vec`, the slicer traces the `self` argument back through the path:
- `self` → the `NonNull<Node>` from `Box::leak(node)`
- `node` → the `Box::new(Node { ... })`
- → `ContractFact`: nothing relevant here (constructor has no `requires`)

The slicer classifies items as `Statement` (MIR statements in the def-use chain), `Terminator` (control-flow decisions), `ContractFact` (caller's `#[rapx::requires]`), or `Forget` (facts invalidated by unknown calls).

## 8.4.3 Forward Verification

The `ForwardVerifier` walks forward along the retained items, building abstract state:

- **`values`**: Maps MIR locals to `AbstractValue` — Place, Ref, RawPtr, ConstInt, etc.
- **`facts`**: `PointsTo`, `KnownAligned`, `KnownInit`, `KnownAllocated`, `KnownNonZero`, `BranchEq`, and `CallEffect`.

For `from_vec`'s `as_mut` callsite, the forward visit traces `node` → `Box::new` → `Box::leak` → `NonNull::from`. The `ReturnPointerFromArg` call effect from `Box::leak` establishes that the `NonNull` points to the `Box`'s allocation, and `NonNull::from` preserves this provenance.

## 8.4.4 SMT Check

Each property kind has a dedicated checker module that encodes forward facts and the target property as SMT-LIB assertions, then invokes Z3:

| Module | Checked property |
|--------|-----------------|
| `valid_ptr.rs` | Pointer validity |
| `align.rs` | Pointer alignment |
| `in_bound.rs` | Pointer arithmetic / slice index bounds |
| `init.rs` | Memory initialization |
| `non_null.rs` | Pointer non-null |
| `allocated.rs` | Allocation tracking |
| `owning.rs` | Unique ownership |
| `alias.rs` | Alias hazards |
| `deref.rs` | Composite `Allocated` + `InBound` |
| `typed.rs` | Valid bit-pattern |
| `ptr2ref.rs` | Pointer-to-reference conversion |
| `non_overlap.rs` | Memory region disjointness |
| `alive.rs` | Allocation liveness |
| `valid_transmute.rs` | Transmute validity |
| `transmute_without_align.rs` | Transmute without alignment |
| `valid_num.rs` | Numeric constraints |
| `valid_cstr.rs` | C string validity |
| `non_volatile.rs` | Non-volatile memory |

Properties without a dedicated module produce `Unknown` and do not block soundness.

For the `Ptr2Ref` check in `from_vec`, the `ptr2ref.rs` module verifies that the `NonNull<Node>` was created from a valid `Box` allocation with correct alignment — the forward facts prove this, and Z3 confirms.

For the `Allocated` / `Owning` checks in `drop`, the `allocated.rs` and `owning.rs` modules verify against facts derived from the struct invariant (see [§8.4.6](#846-struct-invariant-verification)). Since the invariant declares `Allocated(head.unwrap_some(), Node, 1)` and `Owning(head.unwrap_some())`, and `head` is `self.head` in `drop`, the facts are sufficient.

## 8.4.5 Call Effect Summaries

When the verifier encounters a call to a standard library function, it looks up the function's effects in a curated database rather than inlining the callee's MIR:

| Effect | Example |
|--------|---------|
| `ReturnPointerFromArg` | Return aliases an argument | `NonNull::as_ptr` |
| `ReturnPointerAdd` | Return = base + offset × stride | `ptr::add` |
| `ReturnNonZero` | Return is non-zero | `NonNull::new_unchecked` |
| `ReturnAligned` | Return is aligned | `alloc::alloc` |
| `ReturnConst` | Return is a constant | `size_of`, `align_of` |
| `ReturnLengthOfArg` | Return = len(arg) | `slice::len` |
| `ReadMemory` | Reads from pointed-to region | `ptr::read` |
| `WriteMemory` | Writes (establishes `KnownInit`) | `ptr::write` |
| `ForgetArgFacts` | Invalidates facts about an arg | `ptr::swap` |
| `OwnsInitMemory` | Callee takes ownership | `Box::from_raw` |

In `from_vec`, `Box::leak` has a `ReturnPointerFromArg` effect, telling the verifier that the `NonNull` returned by `from(leak(node))` points to the same allocation as the `Box`. In `drop`, `Box::from_raw` has an `OwnsInitMemory` effect, telling the verifier it takes ownership of the raw pointer — establishing the `Owning` fact needed for subsequent checks.

## 8.4.6 Struct Invariant Verification

For methods on a struct with `#[rapx::invariant]`, the verifier additionally checks invariants at every observable point:

- **Constructors** (returning `Self`): Invariants must be proved at each return checkpoint.
- **Methods** (`&self` / `&mut self`): Invariants are assumed at entry and verified at all return points.

In `from_vec`, the invariants are checked at `bb15` (the return block). Three paths reach this point — the initial construction (`path 0, 1, 2, 3, 6, 15` where the loop never executes) and two loop-body paths. The verifier proves that `head` (when `Some`) satisfies `Align`, `Allocated`, and `Owning` along all three.

In `drop`, invariants are checked at `bb7` (function entry, the loop-exit path) and `bb8` (after each `from_raw` call). The invariant facts are present at entry because `drop` receives `&mut self` with the struct's invariants intact.

### Discharging Obligations from Invariants

When a pointer is obtained from a struct field with a matching invariant, the verifier automatically discharges related obligations. For `drop`'s `Box::from_raw` call, the verifier needs `Allocated(raw, T, 1)` and `Owning(raw)` for the raw pointer obtained from `head.unwrap_some()`. Since the `LinkedList` invariant already declares these properties, the obligations are discharged without requiring explicit annotations.

This discharge mechanism (`discharge_from_field_invariant` in `smt_check/field_invariant.rs`) traces the pointer backward through up to 32 value-copy steps, following assignments, casts, and pointer-returning calls. Four property kinds are discharged this way:

| Property | Example Invariant |
|----------|------------------|
| `Owning` | `#[rapx::invariant(Owning(field.unwrap_some()))]` |
| `Allocated` | `#[rapx::invariant(Allocated(field, T, n))]` |
| `ValidPtr` | `#[rapx::invariant(ValidPtr(field, T, n))]` |
| `Typed` | `#[rapx::invariant(Typed(field, T))]` |

Two cross-kind implication rules from `primitive-sp.md` are also applied:

- **`Init ⇒ Typed`**: An `Init(field, T, count)` invariant discharges `Typed(field, T)` at a dereference.
- **`ValidPtr ⇒ Allocated ∧ InBound`** (non-ZST): Discharges both allocation and bounds checks.

These allow concise invariants — `#[rapx::invariant(Init(ptr, T, cap))]` alone suffices for `Typed` checks.

## 8.4.7 Invless Mode

In `invless` mode, the verifier generates constructor-to-method sequences when struct invariants are not annotated:

```
Constructor → Method
Constructor → Mutator → Method
```

Each sequence propagates the constructor's `#[rapx::requires]` through the mutator chain, filtering out contracts invalidated by field mutations. The accumulated contracts are then checked against each method's unsafe call sites.

`invless` mode helps identify where explicit invariants are needed: an unproved sequence indicates either insufficient constructor contracts or a mutator invalidating a contract the reader depends on.

## 8.4.8 Raw Pointer Dereference Checking

Direct `*ptr` and `*ptr = val` in unsafe blocks are treated as implicit checkpoints:

- **Reads** (`*ptr`): Requires `ValidPtr(ptr, T, 1)`, `Align(ptr, T)`, `Typed(ptr, T)`.
- **Writes** (`*ptr = val`): Requires `ValidPtr(ptr, T, 1)`, `Align(ptr, T)`.

These use the same pipeline and benefit from path sensitivity and abstract state tracking.

## 8.4.9 Static Mut Access Checking

`static mut` accesses generate implicit safety contracts for each access point:
- **Reads and writes**: `ValidPtr(ptr, T, 1)`, `Align(ptr, T)`
- **Reads additionally**: `Init(ptr, T, 1)` — the static must have been written before reading.
