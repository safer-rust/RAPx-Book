# Chapter 8.4. The Verification Pipeline

This chapter walks through the verification pipeline using the [`linked_list_nonnull`](https://github.com/safer-rust/RAPx/blob/main/rapx/tests/verify_cases/linked_list_nonnull/src/lib.rs) case study — a doubly-linked list built with `NonNull<Node>` pointers:

```rust
#[rapx::invariant(Align(head.unwrap_some(), Node))]
#[rapx::invariant(Allocated(head.unwrap_some(), Node, 1))]
#[rapx::invariant(Owning(head.unwrap_some()))]
struct LinkedList { head: Option<NonNull<Node>>, tail: Option<NonNull<Node>> }
```

The report shows two verification targets, each with its own contracts to check:

| Target | What to verify |
|--------|---------------|
| `LinkedList::from_vec` (constructor) | struct invariants at return (`Align`, `Allocated`, `Owning`); `Ptr2Ref` at calls to `NonNull::as_mut` |
| `<LinkedList as Drop>::drop` (destructor) | struct invariants at exit; `Ptr2Ref` at `NonNull::as_ref`; `Allocated`/`Owning`/`Alias` at `Box::from_raw` |

For each target, the pipeline executes the same sequence: **path extraction** (§8.4.1), **backward slicing** (§8.4.2), **forward verification** (§8.4.3), and **SMT check** (§8.4.4). The sections below walk through each step using examples from both targets.

## 8.4.1 Path Extraction

The first stage enumerates acyclic paths from function entry to each unsafe callsite using `PathGraph`'s SCC decomposition (see [§5.1 Path Analysis](./5.1-path.md) for the full algorithm). Each path is a sequence of basic block IDs — the route the verifier follows when tracking state.

Take `LinkedList::from_vec` as the running example. Its body iterates over `values`, pushing each element onto the list. The loop has two branches: the `tail` is `None` (first iteration) or `Some` (subsequent iterations). The path extractor unrolls this loop into four distinct paths reaching the two callsites:

```
unsafe checkpoint: bb10 -> core::ptr::non_null::as_mut
  ┌─ path 0 (first iteration, tail is None):
  │   0 → 1 → 2 → 3 → 5 → 7 → 8 → 9 → 11 → 14 → 2 → 3 → … → 10
  │   Ptr2Ref | Proved
  └─ path 1 (subsequent iteration, tail is Some):
      0 → 1 → … → 14 → 2 → 3 → 5 → … → 10 (shorter: no tail init)
      Ptr2Ref | Proved

unsafe checkpoint: bb12 -> core::ptr::non_null::as_mut
  ┌─ path 0 (first iteration → second push):
  │   0 → … → 10 → 12 → 13 → 14 → 2 → 3 → … → 10 → 12
  │   Ptr2Ref | Proved
  └─ path 1 (subsequent iteration → second push):
      0 → … → 10 → 12 (simpler: tail already set)
      Ptr2Ref | Proved
```

Why four paths? The SCC containing the `for` loop body is unrolled once. Each unrolling explores both branches (`None`/`Some`), producing two entry paths (first iteration) and two repeat paths (subsequent iterations). The verifier checks `Ptr2Ref` on every path to ensure the `NonNull<Node>` can always be soundly converted to a reference — whether this is the first node added to an empty list or a subsequent one.

After path extraction, the struct invariant checkpoints at the return block (bb15) are enumerated similarly, yielding three reaching paths: the empty-vec path (`0 → 1 → 2 → 3 → 6 → 15`) and two loop-body paths (one from each branch).

Paths are stored in a `PathTree` per callsite and capped at 1024. By default (`--postfix-repeat` not set), verification starts at `repeat = 0` — each SCC body appears at most once per path. After the first round, **auto-detectors** (see [§8.4.7](#847-auto-repeat-detectors)) decide whether deeper unrolling is needed for loop-sensitive property kinds. In `from_vec`, all paths are covered at repeat 0, so no auto-expansion occurs.

## 8.4.2 Backward Slicing

First, inspect the actual MIR with `cargo rapx analyze mir` to see the basic blocks and locals:

```
bb5: {
    _17 = Box::<Node>::new(move _18) → [return: bb7, unwind: bb16]
}
bb7: {
    _24 = move _17
    _23 = Box::<Node>::leak(move _24) → [return: bb8, unwind: bb16]
}
bb8: {
    _22 = NonNull::<Node>::from(move _23) → [return: bb9, unwind: bb16]
}
bb9: {
    _25 = discriminant((_2.1: Option<NonNull<Node>>))
    switchInt(move _25) → [0: bb11, 1: bb10]
}
bb10: {
    _30 = copy ((_2.1 as Some).0: NonNull<Node>)
    _34 = &mut _30
    _33 = NonNull::<Node>::as_mut(move _34) → [return: bb12, unwind: bb16]
}
bb12: {
    ((*_33).2: Option<NonNull<Node>>) = move _31
    _38 = &mut _22
    _37 = NonNull::<Node>::as_mut(move _38) → [return: bb13, unwind: bb16]
}
```

The `from_vec` MIR has two `as_mut` callsites — bb10 (`tail.as_mut()`) and bb12 (`node.as_mut()`). The `BackwardSlicer` walks backward from each checkpoint, collecting only the statements whose value flow reaches the target.

For the `Ptr2Ref` check at **bb12** (`node.as_mut()`), the target place is `_38` — a `&mut NonNull<Node>`. The slicer traces the value chain:

| Block | Statement | Slicer action |
|-------|-----------|---------------|
| bb12 | `_38 = &mut _22` | Reference to `_22` — follow `_22`'s definition |
| bb12 | `_22` is a local | Defined at bb8 — include `NonNull::from(move _23)` |
| bb8 | `_22 = NonNull::from(move _23)` | `_23` is a `&mut Node` from `Box::leak` — include the call |
| bb7 | `_23 = Box::<Node>::leak(move _24)` | `_24` aliases `_17` (the Box) — follow to `Box::new` |
| bb5 | `_17 = Box::<Node>::new(move _18)` | `Rvalue::ShallowInitBox` — allocation origin — **stop** |

The collected trace for this path is four statements: `Box::new` → `Box::leak` → `NonNull::from` → `&mut _22`. The slicer drops unrelated statements (the `for` loop iterator, value reads, `match` pattern bindings) because they don't affect `_22`'s provenance.

For the checkpoint at **bb10** (`tail.as_mut()`), the target is `_34` (`&mut _30`), which aliases `(_2.1 as Some).0` — the `NonNull` stored in `list.tail`. The slicer traces the same allocation chain through `_22`'s prior assignment at bb8. Same facts, different access path.

For struct invariant return checkpoints at **bb15**, the slicer walks backward from `_0 = move _2` (the returned `LinkedList`). It collects `_2`'s construction at bb0 (empty `head`/`tail`) and the assignments at bb11/bb13 (updates to `_2.0` and `_2.1`), linking back to the same `Box::new` → `NonNull::from` chain for the non-empty cases.

No `ContractFact` items are injected because `from_vec` has no `#[rapx::requires]` of its own.

## 8.4.3 Forward Verification

The `ForwardVerifier` walks forward along the sliced trace from §8.4.2, building abstract state. Follow the **bb12 path** (`node.as_mut()`) as the example: `bb5 → bb7 → bb8 → bb9 → bb12`.

**bb5** — `Box::new(Node { … })`: `_17 = Box::<Node>::new(move _18)` is an `Rvalue::ShallowInitBox`. Two facts are emitted:

| Fact | Meaning |
|------|---------|
| `KnownNonZero(_17)` | Box pointer is non-null |
| `KnownAllocated(_17, Box<Node>, 1)` | `_17` points to a live heap allocation |

**bb7** — `Box::leak`: `_24 = move _17` copies the Box, then `_23 = Box::<Node>::leak(move _24)` transfers ownership — the Box is consumed, a `&mut Node` returned. The verifier tracks the value flow from `_17` through `_24` to `_23`.

**bb8** — `NonNull::from`: `_22 = NonNull::<Node>::from(move _23)` applies the `ReturnPointerFromArg` effect — the return value's pointer aliases argument `_23`, which traces back to the Box allocation `_17`. The verifier records `PointsTo(_22, _17)`.

**bb9** — branch: `switchInt(move _25)` on `(_2.1)`'s discriminant. The `Some` branch (tag = 1) is selected — a `PathCondition` records `_25 = 1`.

**bb12** — checkpoint: `_38 = &mut _22` creates a reference, the verifier records `PointsTo(_38, _22)`. Then `_37 = NonNull::as_mut(move _38)` is the `Ptr2Ref` checkpoint. The checker resolves `_38 → _22 → _17`, finds `KnownNonZero(_17)` and `KnownAllocated(_17, Box<Node>, 1)`, and proves non-null + aligned. Result: `Ptr2Ref | Proved`.

The struct invariants at `bb15` follow the same value chain — `head.unwrap_some()` resolves through the `_2.0` field back to `_17`, and the same allocation facts discharge all three invariant checks (`Align`, `Allocated`, `Owning`), yielding `SOUND`.

## 8.4.4 SMT Check

Continuing the bb12 path from §8.4.3, the forward facts are encoded into SMT assertions and discharged by the `Ptr2Ref` checker (`ptr2ref.rs`).

The checker receives the checkpoint and resolves the provenance chain:

```
_38 → _22  via  PointsTo(_38, _22)    (the &mut ref points to the NonNull)
_22 → _17  via  PointsTo(_22, _17)    (the NonNull points to the Box allocation)
```

It queries the fact database for `_17`:

```
KnownNonZero(_17)                  →  _17 ≠ 0
KnownAllocated(_17, Box<Node>, 1)  →  alloc_begin(_17) ≠ null
```

The SMT obligation is `(_17 ≠ 0) ∧ (_17 % align_of(Node) == 0)`. The first conjunct falls out of `KnownNonZero`. The second is discharged by `KnownAllocated` — a heap-allocated Box is always correctly aligned. Z3 confirms both trivially: `Ptr2Ref | Proved`.

The struct invariant checks at bb15 operate the same way — `Align`, `Allocated`, and `Owning` all resolve through the same `PointsTo` chain to `_17`, and the same facts discharge them. Each checker module (`align.rs`, `allocated.rs`, `owning.rs`) encodes its specific obligation from the shared fact set.

## 8.4.5 Struct Invariant Verification

Struct invariants are checked at every observable point. For this example, the report shows:

```
--- struct invariants ---
    checkpoint bb15:                    ← from_vec return
      path 0, 1, 2, 3, 6, 15:          ← loop never executes (empty vec)
        Align | Proved
        Allocated | Proved
        Owning | Proved
```

- `from_vec` is a constructor (returns `Self`): invariants are proved at `bb15` along all three reaching paths — the initial construction and two loop-body paths.
- `drop` is a method (`&mut self`): invariants are assumed at entry and checked at `bb7` (loop exit, where the list is empty) and `bb8` (after each `from_raw` call, where the head node has been freed and advanced).
- At `bb7`, the list is empty (`head` is `None`), so `head.unwrap_some()` vacuously satisfies `Align`, `Allocated`, and `Owning` since `Option::None` has no pointer to check.
- At `bb8`, invariants are re-checked after advancing the head pointer in the while loop. The check passes because the invariants track `Allocated` and `Owning` per-node, and the invariant discharge mechanism (below) proves these from the struct's own annotations.

### Discharge from Invariants

When `drop` calls `Box::from_raw(head.unwrap_some().as_ptr())`, the verifier needs `Allocated(raw, T, 1)` and `Owning(raw)`. Instead of requiring explicit annotations at the callsite, the verifier discharges these from the struct invariant:

```
#[rapx::invariant(Allocated(head.unwrap_some(), Node, 1))]
#[rapx::invariant(Owning(head.unwrap_some()))
```

The `discharge_from_field_invariant` mechanism traces the raw pointer back through up to 32 value-copy steps, finds the matching invariant on `LinkedList`, and discharges the obligation. This works for `Owning`, `Allocated`, `ValidPtr`, and `Typed` properties.

Two cross-kind implication rules from `primitive-sp.md` also apply:
- `Init ⇒ Typed`: `Init(field, T, count)` invariant discharges `Typed(field, T)`.
- `ValidPtr ⇒ Allocated ∧ InBound` (non-ZST): Discharges allocation and bounds checks.

## 8.4.6 Call Effect Summaries

Instead of inlining standard library MIR, the verifier uses a curated database of call effects. These are the building blocks that the slicer and forward verifier rely on — each effect provides a fact or value-flow link without expanding the callee's body.

| Effect | How it works |
|--------|-------------|
| `ReturnPointerFromArg` | The return value is a pointer extracted from an argument (e.g. `NonNull::from`). |
| `BoxAllocation` | A `Box::new` call produces a fresh heap allocation. |
| `OwnsInitMemory` | A `Box::from_raw` call takes ownership of a raw pointer. |
| `ReturnConst` | The return value is a compile-time constant (`align_of`, `size_of`). |

In `from_vec`, the slicer captures `Box::new` (→ `KnownAllocated`) and `NonNull::from` (→ `PointsTo` linking the return value to the allocation). The forward verifier then carries these facts to the checkpoint.


## 8.4.7 Auto-Repeat Detectors

Loop-carried properties (bounds checks, heap allocations) sometimes need more than one loop iteration to verify. The verifier starts at `postfix_repeat = 0` (each SCC body appears at most once). After the first round, dedicated auto-detectors run to decide whether deeper unrolling would help.

### Phase 1.5 — InBound

The InBound detector addresses **off-by-one errors**. At repeat 0, a bounds check inside a loop may appear `Proved` because the path only exercises the first iteration. A bug that triggers on the second iteration (e.g. `index == len`) would be missed.

- **Trigger**: all InBound results are `Proved` at the current depth.
- **Action**: progressively unroll up to 16 levels. At each level, re-run verification.
- **Stop conditions**: an InBound check becomes `Unproved` (found a bug — report it); or the `Proved` set stabilises (same as a previous level); or the `Proved` set oscillates between two states.

### Phase 1.6 — Deref / ValidPtr

Same logic as InBound, applied to `Deref` and `ValidPtr` results. These are compound properties (`Allocated ∧ InBound`) that depend on loop-carried allocation and bounds facts.

### Phase 1.7 — Allocated / Deref / ValidPtr

This detector handles **allocation reachability**. Fresh allocations inside loops (e.g. `Box::new` in a `for` body) produce facts (`PointsTo`, `KnownAllocated`) that the slicer needs to reach. The problem: at repeat 0 the allocation site appears on the path, but the fact may be too shallow in the trace for the forward verifier to carry it all the way to the callsite. Deeper unrolling repeats the loop body, extending the trace so the slicer can bridge the gap.

- **Trigger**: any Allocated / Deref / ValidPtr result is `Unproved` (excluding precision loss / connectivity failures).
- **Action**: retry with progressively deeper unrolling (up to 16). A new `VerifyDriver` is spawned at each level.
- **Stop conditions**: all targeted properties become `Proved`; or the set of still-unproved results converges to a previous level; or it oscillates.

In all three phases, earlier results from lower repeat levels are retained. Only newly `Proved` results from deeper levels are merged in — the mechanism is monotonic and never weakens a verdict.
