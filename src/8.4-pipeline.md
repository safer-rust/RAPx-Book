# Chapter 8.4. The Verification Pipeline

This chapter walks through the verification pipeline using the [`linked_list_nonnull`](https://github.com/safer-rust/RAPx/blob/main/rapx/tests/verify_cases/linked_list_nonnull/src/lib.rs) case study — a doubly-linked list built with `NonNull<Node>` pointers:

```rust
#[rapx::invariant(Align(head.unwrap_some(), Node))]
#[rapx::invariant(Allocated(head.unwrap_some(), Node, 1))]
#[rapx::invariant(Owning(head.unwrap_some()))]
struct LinkedList { head: Option<NonNull<Node>>, tail: Option<NonNull<Node>> }
```

The report shows two verification targets, each with its own contracts to check:

| Target | What to verify |
|--------|---------------|
| `LinkedList::from_vec` (constructor) | struct invariants at return (`Align`, `Allocated`, `Owning`); `Ptr2Ref` at calls to `NonNull::as_mut` |
| `<LinkedList as Drop>::drop` (destructor) | struct invariants at exit; `Ptr2Ref` at `NonNull::as_ref`; `Allocated`/`Owning`/`Alias` at `Box::from_raw` |

For each target, the pipeline executes the same sequence: **path extraction** (§8.4.1), **backward slicing** (§8.4.2), **forward verification** (§8.4.3), and **SMT check** (§8.4.4). The sections below walk through each step using examples from both targets.

## 8.4.1 Path Extraction

The first stage enumerates acyclic paths from function entry to each unsafe callsite using `PathGraph`'s SCC decomposition (see [§5.1 Path Analysis](./5.1-path.md) for the full algorithm). Each path is a sequence of basic block IDs — the route the verifier follows when tracking state.

Take `LinkedList::from_vec` as the running example. Its body iterates over `values`, pushing each element onto the list. The loop has two branches: the `tail` is `None` (first iteration) or `Some` (subsequent iterations). The path extractor unrolls this loop into four distinct paths reaching the two callsites:

```
unsafe checkpoint: bb10 -> core::ptr::non_null::as_mut
  ┌─ path 0 (first iteration, tail is None):
  │   0 → 1 → 2 → 3 → 5 → 7 → 8 → 9 → 11 → 14 → 2 → 3 → … → 10
  │   Ptr2Ref | Proved
  └─ path 1 (subsequent iteration, tail is Some):
      0 → 1 → … → 14 → 2 → 3 → 5 → … → 10 (shorter: no tail init)
      Ptr2Ref | Proved

unsafe checkpoint: bb12 -> core::ptr::non_null::as_mut
  ┌─ path 0 (first iteration → second push):
  │   0 → … → 10 → 12 → 13 → 14 → 2 → 3 → … → 10 → 12
  │   Ptr2Ref | Proved
  └─ path 1 (subsequent iteration → second push):
      0 → … → 10 → 12 (simpler: tail already set)
      Ptr2Ref | Proved
```

Why four paths? The SCC containing the `for` loop body is unrolled once. Each unrolling explores both branches (`None`/`Some`), producing two entry paths (first iteration) and two repeat paths (subsequent iterations). The verifier checks `Ptr2Ref` on every path to ensure the `NonNull<Node>` can always be soundly converted to a reference — whether this is the first node added to an empty list or a subsequent one.

After path extraction, the struct invariant checkpoints at the return block (bb15) are enumerated similarly, yielding three reaching paths: the empty-vec path (`0 → 1 → 2 → 3 → 6 → 15`) and two loop-body paths (one from each branch).

Paths are stored in a `PathTree` per callsite and capped at 1024. When `--postfix-repeat` is not explicitly set, the verifier auto-expands postfix repetitions until convergence per property phase.

## 8.4.2 Backward Slicing

For each path, the `BackwardSlicer` walks backward from the callsite, collecting only MIR statements and terminators relevant to the current property's target place.

Take the `Ptr2Ref` check at `as_mut` in `from_vec` (line 20). The property target is the `self` argument — the `NonNull<Node>` obtained from `Box::leak(node)`. The slicer traces back through:

- `as_mut(self)` ← `self` is a `&mut NonNull<Node>`
- The `NonNull<Node>` comes from `NonNull::from(Box::leak(node))`
- `Box::leak(node)` returns `&mut Node` — this is the `ReturnPointerFromArg` effect
- `node` comes from `Box::new(Node { … })` — a fresh allocation

The collected items include the `Box::new` constructor call (providing allocation facts) and the `Box::leak` → `NonNull::from` chain (providing provenance). No `ContractFact` items are injected because `from_vec` has no `#[rapx::requires]` of its own.

## 8.4.3 Forward Verification

The `ForwardVerifier` walks forward along the retained path items, building abstract state:

- At `Box::new(Node { … })`: a `KnownAllocated` fact is recorded for the new `Box<Node>`.
- At `Box::leak(node)`: the callee summary provides `ReturnPointerFromArg` — the `NonNull` points into the `Box`'s allocation. A `PointsTo` fact links the raw pointer (inside `NonNull`) to the allocation.
- At `NonNull::from(leak_result)`: the `NonNull`'s internal pointer inherits the `RawPtr` provenance.
- At `as_mut(&mut non_null)`: the `Ptr2Ref` checker finds `PointsTo` linking the `NonNull`'s pointer to a known allocation. Combined with the alignment type (`Node`), the obligation is satisfied.

For `drop`'s `from_raw` call, the forward visit is simpler: the invariant facts `Allocated(head.unwrap_some(), Node, 1)` and `Owning(head.unwrap_some())` are available at entry because `drop` takes `&mut self`. The forward visit just propagates these facts to the `from_raw` callsite — no computation needed.

## 8.4.4 SMT Check

The SMT check encodes forward facts plus the target property into SMT-LIB assertions and invokes Z3. Each property kind maps to a dedicated checker module:

| Module | What it checks | Used in this example? |
|--------|---------------|----------------------|
| `ptr2ref.rs` | Pointer-to-reference conversion | `as_mut` and `as_ref` calls |
| `allocated.rs` | Memory allocation tracking | `from_raw` call in `drop` |
| `owning.rs` | Unique ownership | `from_raw` call in `drop` |
| `alias.rs` | Alias hazard detection | `from_raw` call in `drop` |
| `align.rs` | Pointer alignment | Struct invariants at return points |
| `valid_ptr.rs` | Pointer validity | (not exercised here) |

For `Ptr2Ref` at `as_mut` in `from_vec`: the checker finds that the `NonNull` was freshly created from a valid `Box` allocation and is properly aligned for `Node`. Z3 confirms both conditions, producing `Proved`.

For the `[hazard] Alias` at `from_raw` in `drop`: the checker confirms that the raw pointer aliases the return value of `Box::from_raw` — this is expected and correct. The `[hazard]` prefix means that even if this were `Unproved`, it would not block the `SOUND` result.

For struct invariants at return points, the checker verifies that `head.unwrap_some()` (when `Some`) satisfies `Align`, `Allocated`, and `Owning` against the forward facts established during construction.

## 8.4.5 Call Effect Summaries

Instead of inlining standard library MIR, the verifier uses a curated database of call effects:

| Effect | Usage in this example |
|--------|----------------------|
| `ReturnPointerFromArg` | `Box::leak` — the returned pointer aliases the Box's allocation |
| `OwnsInitMemory` | `Box::from_raw` — takes ownership of the raw pointer |
| `ReturnConst` | `NonNull::from` — wraps a raw pointer, preserving provenance |
| `ReturnPointerAdd` | (not used here) |
| `ReadMemory` | (not used here) |

In `from_vec`: `Box::leak(node)` → `NonNull::from(…)` → stored in `list.head`. The `ReturnPointerFromArg` effect tells the verifier that `list.head.unwrap_some()` points to a live, owned allocation. This fact is later used at the struct invariant return checkpoints.

In `drop`: `Box::from_raw(node.as_ptr())` → the `OwnsInitMemory` effect tells the verifier that the `Box` reclaims ownership, which pairs with the `Owning` invariant declared on the struct.

## 8.4.6 Struct Invariant Verification

Struct invariants are checked at every observable point. For this example, the report shows:

```
--- struct invariants ---
    checkpoint bb15:                    ← from_vec return
      path 0, 1, 2, 3, 6, 15:          ← loop never executes (empty vec)
        Align | Proved
        Allocated | Proved
        Owning | Proved
```

- `from_vec` is a constructor (returns `Self`): invariants are proved at `bb15` along all three reaching paths — the initial construction and two loop-body paths.
- `drop` is a method (`&mut self`): invariants are assumed at entry and checked at `bb7` (loop exit, where the list is empty) and `bb8` (after each `from_raw` call, where the head node has been freed and advanced).
- At `bb7`, the list is empty (`head` is `None`), so `head.unwrap_some()` vacuously satisfies `Align`, `Allocated`, and `Owning` since `Option::None` has no pointer to check.
- At `bb8`, invariants are re-checked after advancing the head pointer in the while loop. The check passes because the invariants track `Allocated` and `Owning` per-node, and the invariant discharge mechanism (below) proves these from the struct's own annotations.

### Discharge from Invariants

When `drop` calls `Box::from_raw(head.unwrap_some().as_ptr())`, the verifier needs `Allocated(raw, T, 1)` and `Owning(raw)`. Instead of requiring explicit annotations at the callsite, the verifier discharges these from the struct invariant:

```
#[rapx::invariant(Allocated(head.unwrap_some(), Node, 1))]
#[rapx::invariant(Owning(head.unwrap_some()))
```

The `discharge_from_field_invariant` mechanism traces the raw pointer back through up to 32 value-copy steps, finds the matching invariant on `LinkedList`, and discharges the obligation. This works for `Owning`, `Allocated`, `ValidPtr`, and `Typed` properties.

Two cross-kind implication rules from `primitive-sp.md` also apply:
- `Init ⇒ Typed`: `Init(field, T, count)` invariant discharges `Typed(field, T)`.
- `ValidPtr ⇒ Allocated ∧ InBound` (non-ZST): Discharges allocation and bounds checks.

## 8.4.7 Invless Mode

When struct invariants are not annotated, `invless` mode generates constructor-to-method sequences instead:

```
Constructor → Method
Constructor → Mutator → Method
```

The `linked_list_nonnull` example uses explicit invariants, so invless mode is not needed. Invless mode helps identify where invariants should be added: an unproved sequence indicates that a constructor's contracts are insufficient or a mutator invalidates a contract.

## 8.4.8 Raw Pointer Dereference

Direct `*ptr` and `*ptr = val` in unsafe blocks are treated as implicit checkpoints requiring `ValidPtr`, `Align`, and (for reads) `Typed`. The `linked_list_nonnull` example works entirely through safe abstractions (`NonNull`, `Option`, `Box`) and unsafe function calls (`as_mut`, `as_ref`, `from_raw`), so no raw pointer dereference checkpoints appear in its report.

## 8.4.9 Static Mut Access

`static mut` accesses generate implicit safety contracts requiring `ValidPtr`, `Align`, and (for reads) `Init`. The example does not use `static mut`, so no such checkpoints appear.
