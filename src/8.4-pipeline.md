# Chapter 8.4. The Verification Pipeline

This chapter walks through the verification pipeline using the [`linked_list_nonnull`](https://github.com/safer-rust/RAPx/blob/main/rapx/tests/verify_cases/linked_list_nonnull/src/lib.rs) case study — a doubly-linked list built with `NonNull<Node>` pointers:

```rust
#[rapx::invariant(Align(head.unwrap_some(), Node))]
#[rapx::invariant(Allocated(head.unwrap_some(), Node, 1))]
#[rapx::invariant(Owning(head.unwrap_some()))]
struct LinkedList { head: Option<NonNull<Node>>, tail: Option<NonNull<Node>> }
```

The report shows two verification targets, each with its own contracts to check:

| Target | What to verify |
|--------|---------------|
| `LinkedList::from_vec` (constructor) | struct invariants at return (`Align`, `Allocated`, `Owning`); `Ptr2Ref` at calls to `NonNull::as_mut` |
| `<LinkedList as Drop>::drop` (destructor) | struct invariants at exit; `Ptr2Ref` at `NonNull::as_ref`; `Allocated`/`Owning`/`Alias` at `Box::from_raw` |

For each target, the pipeline executes the same sequence: **path extraction** (§8.4.1), **backward slicing** (§8.4.2), **forward verification** (§8.4.3), and **SMT check** (§8.4.4). The sections below walk through each step using examples from both targets.

## 8.4.1 Path Extraction

The first stage enumerates acyclic paths from function entry to each unsafe callsite using `PathGraph`'s SCC decomposition (see [§5.1 Path Analysis](./5.1-path.md) for the full algorithm). Each path is a sequence of basic block IDs — the route the verifier follows when tracking state.

Take `LinkedList::from_vec` as the running example. Its body iterates over `values`, pushing each element onto the list. The loop has two branches: the `tail` is `None` (first iteration) or `Some` (subsequent iterations). The path extractor unrolls this loop into four distinct paths reaching the two callsites:

```
unsafe checkpoint: bb10 -> core::ptr::non_null::as_mut
  ┌─ path 0 (first iteration, tail is None):
  │   0 → 1 → 2 → 3 → 5 → 7 → 8 → 9 → 11 → 14 → 2 → 3 → … → 10
  │   Ptr2Ref | Proved
  └─ path 1 (subsequent iteration, tail is Some):
      0 → 1 → … → 14 → 2 → 3 → 5 → … → 10 (shorter: no tail init)
      Ptr2Ref | Proved

unsafe checkpoint: bb12 -> core::ptr::non_null::as_mut
  ┌─ path 0 (first iteration → second push):
  │   0 → … → 10 → 12 → 13 → 14 → 2 → 3 → … → 10 → 12
  │   Ptr2Ref | Proved
  └─ path 1 (subsequent iteration → second push):
      0 → … → 10 → 12 (simpler: tail already set)
      Ptr2Ref | Proved
```

Why four paths? The SCC containing the `for` loop body is unrolled once. Each unrolling explores both branches (`None`/`Some`), producing two entry paths (first iteration) and two repeat paths (subsequent iterations). The verifier checks `Ptr2Ref` on every path to ensure the `NonNull<Node>` can always be soundly converted to a reference — whether this is the first node added to an empty list or a subsequent one.

After path extraction, the struct invariant checkpoints at the return block (bb15) are enumerated similarly, yielding three reaching paths: the empty-vec path (`0 → 1 → 2 → 3 → 6 → 15`) and two loop-body paths (one from each branch).

Paths are stored in a `PathTree` per callsite and capped at 1024. By default (`--postfix-repeat` not set), verification starts at `repeat = 0` — each SCC body appears at most once per path. After the first round, **auto-detectors** (see [§8.4.10](#8410-auto-repeat-detectors)) decide whether deeper unrolling is needed for loop-sensitive property kinds. In `from_vec`, all paths are covered at repeat 0, so no auto-expansion occurs.

## 8.4.2 Backward Slicing

For each path, the `BackwardSlicer` walks backward from the callsite, collecting only the MIR statements and terminators relevant to the current property's target place. This prunes the path to a minimal trace, keeping the forward verifier focused on value-flow chains that matter.

Take the `Ptr2Ref` check at `tail.as_mut()` (path `0 → … → 10`) as the running example. The return type of `as_mut` is `&mut Node`, so the slicer identifies `PlaceKey::ReturnPointer` as the target and walks backward through the MIR:

1. **`as_mut` call (bb10):** the argument is `_12: &mut NonNull<Node>`, the reference to the `NonNull` stored in `list.tail`.
2. **Store to `list.tail` (bb8):** `_12` refers to `_9` which was set to `Some(_3)` at bb8 — the assignment `list.tail = Some(node)`.
3. **`NonNull::from(Box::leak(node))` (bb6):** `_3` (`node`) is the result of `NonNull::from(Box::leak(node))`. The slicer includes the `Box::leak` call (for its value-flow effect) and the `NonNull::from` call (whose `ReturnPointerFromArg` effect links the result to the Box allocation).
4. **`Box::new(Node { … })` (bb5):** `_1` is the result of `Rvalue::ShallowInitBox`, which the verifier recognises as a fresh heap allocation. The slicer collects this statement so the forward verifier can emit `KnownAllocated`.

The collected trace for this path is: `Box::new` → `NonNull::from(Box::leak(…))` → store to tail → `as_mut` call. The slicer drops unrelated operations (the `for` loop iterator advance, value reads) because they don't affect the target place.

For struct invariant return checkpoints (bb15), the slicer walks backward from the return statement and collects the same `Box::new` allocation, the `NonNull::from` wrapper, and the assignment to `list.head` — the facts needed to prove `head.unwrap_some()` satisfies `Align`, `Allocated`, and `Owning`.

No `ContractFact` items are injected because `from_vec` has no `#[rapx::requires]` of its own.

## 8.4.3 Forward Verification

The `ForwardVerifier` walks forward along the sliced trace, building an abstract state by accumulating facts from Rvalues, calls, and branch conditions. Here is the step-by-step state evolution for the `Ptr2Ref` path through the `Some(tail)` branch:

**Step 1 — `Box::new(Node { … })` (bb5)**

The `Rvalue::ShallowInitBox` is encountered. Two facts are emitted:

| Fact | Meaning |
|------|---------|
| `KnownNonZero(_1)` | the `Box<Node>` pointer is non-null (Box is heap-allocated) |
| `KnownAllocated(_1, Box<Node>, 1)` | `_1` points to a live heap allocation |

**Step 2 — `NonNull::from(Box::leak(_1))` (bb6)**

`Box::leak(_1)` transfers ownership: the Box is consumed and a `&mut Node` returned. The verifier tracks this as a value flow through the `_1` local. Then `NonNull::from(&mut Node)` applies the `ReturnPointerFromArg` call effect — the return value inherits the provenance of its argument. The verifier records:

| Fact | Meaning |
|------|---------|
| `PointsTo(_3, _1)` | `_3` (the NonNull wrapper) points to the `Box<Node>` allocation |
| `KnownNonNull(_3)` | inherited from the Box's non-null guarantee (via provenance) |

**Step 3 — Store to `list.tail` (bb8)**

`_9 = Some(_3)` wraps the `NonNull` in an `Option::Some`. The verifier records the assignment in its value-definition table so that later lookups on `_9` resolve to `_3`.

**Step 4 — Enter `unsafe { }` block (bb9) and `match list.tail`**

The branch condition `list.tail` is `Some` is recorded as a `PathCondition`. The `_12 = &mut _9.0` (`&mut list.tail.unwrap_some()`) creates a mutable reference — the verifier records:

| Fact | Meaning |
|------|---------|
| `PointsTo(_12, _3)` | the reference `_12` points to the NonNull `_3` |

**Step 5 — `tail.as_mut()` call (bb10)**

This is the checkpoint. The `Ptr2Ref` checker needs to prove that `_12` (a `&mut NonNull<Node>`) can be soundly converted to `&mut Node`. The checker:

1. Looks up `_12`'s target: resolves through `PointsTo` to `_3`.
2. Looks up `_3`'s provenance: resolves through `PointsTo` to `_1` (the Box allocation).
3. Verifies `_1` is non-null — satisfied by `KnownNonZero`.
4. Infers alignment: the pointee type is `Node`, alignment is 8 bytes. Since `_1` points to a valid Box allocation, the alignment obligation is discharged by the `KnownAllocated` fact.

All checks pass → `Ptr2Ref | Proved`.

**Step 6 — Struct invariant return checkpoint (bb15)**

After all loop iterations, `list` is returned. The verifier checks the struct invariants declared on `LinkedList`:

- `Align(head.unwrap_some(), Node)` — resolves through `PointsTo` to the same Box allocation, alignment satisfied by `KnownAllocated`.
- `Allocated(head.unwrap_some(), Node, 1)` — checks that the NonNull points to a live allocation object of type `Node`. Satisfied by `KnownAllocated` linked through the `PointsTo` chain.
- `Owning(head.unwrap_some())` — checks no other reference aliases the pointee. The verifier's provenance tracking confirms that `_1` was freshly allocated and its ownership was transferred only through `Box::leak` → `NonNull::from` — no other alias exists.

All invariants pass → the function is `SOUND`.

## 8.4.4 SMT Check

Each property kind maps to a dedicated checker module. The forward facts accumulated in §8.4.3 are encoded into SMT-LIB assertions and discharged by the Z3 solver.

**Ptr2Ref at `as_mut`**

The `ptr2ref.rs` checker receives the checkpoint and the target `_12`. It resolves the provenance chain `_12 → _3 → _1` and queries the fact database:

```
KnownNonZero(_1)     →  encodes  _1 ≠ 0
KnownAllocated(_1, Box<Node>, 1)  →  encodes  alloc_begin(_1) ≠ null
```

The SMT obligation is `(_1 ≠ 0) ∧ (_1 % align_of(Node) == 0)`. Z3 confirms both conjuncts from the encoding of the two facts — no solver query needed beyond what's already established in the fact database.

**Allocated at struct return**

The `allocated.rs` checker resolves `head.unwrap_some()` through the `PointsTo` chain to `_1`, then queries `KnownAllocated(_1, Box<Node>, 1)`. The fact records that `_1` is a live allocation of type `Box<Node>` with 1 element. The checker emits the obligation `alloc_begin(_1) ≠ null ∧ alloc_end(_1) = alloc_begin(_1) + sizeof(Node)`. Z3 trivially proves this from the stored fact.

**Owning at struct return**

The `owning.rs` checker looks for conflicting aliases. It scans all `PointsTo` entries and confirms that no other pointer aliases the allocation object of `_1`. The solver obligation `∀p ≠ _1. ¬PointsTo(p, _1)` is vacuous because no other `PointsTo` entry references `_1`'s allocation object.

**Align at struct return**

The `align.rs` checker queries the type layout of `Node` (8-byte aligned) and resolves `head.unwrap_some()` → `_1`. Since `_1` is a Box allocation (heap-allocated with correct alignment), the checker emits a trivial obligation `_1 % 8 == 0` and proves it from the allocation fact.

## 8.4.5 Call Effect Summaries

Instead of inlining standard library MIR, the verifier uses a curated database of call effects:

| Effect | Usage in this example |
|--------|----------------------|
| `ReturnPointerFromArg` | `NonNull::from` — the return value wraps a pointer derived from the argument |
| `OwnsInitMemory` | `Box::from_raw` — takes ownership of the raw pointer |
| `ReturnConst` | (not used in this example — `align_of`, `size_of` calls) |
| `BoxAllocation` | `Box::new` — a fresh heap allocation |

In `from_vec`: `Box::new` → `Box::leak` → `NonNull::from` → stored in `list.head`. The slicer captures `Box::new` (allocation fact) and `NonNull::from` (`ReturnPointerFromArg` effect, linking the NonNull's pointer to the allocation). These facts are later used at the struct invariant return checkpoints.

In `drop`: `Box::from_raw(node.as_ptr())` → the `OwnsInitMemory` effect tells the verifier that the `Box` reclaims ownership, which pairs with the `Owning` invariant declared on the struct.

## 8.4.6 Struct Invariant Verification

Struct invariants are checked at every observable point. For this example, the report shows:

```
--- struct invariants ---
    checkpoint bb15:                    ← from_vec return
      path 0, 1, 2, 3, 6, 15:          ← loop never executes (empty vec)
        Align | Proved
        Allocated | Proved
        Owning | Proved
```

- `from_vec` is a constructor (returns `Self`): invariants are proved at `bb15` along all three reaching paths — the initial construction and two loop-body paths.
- `drop` is a method (`&mut self`): invariants are assumed at entry and checked at `bb7` (loop exit, where the list is empty) and `bb8` (after each `from_raw` call, where the head node has been freed and advanced).
- At `bb7`, the list is empty (`head` is `None`), so `head.unwrap_some()` vacuously satisfies `Align`, `Allocated`, and `Owning` since `Option::None` has no pointer to check.
- At `bb8`, invariants are re-checked after advancing the head pointer in the while loop. The check passes because the invariants track `Allocated` and `Owning` per-node, and the invariant discharge mechanism (below) proves these from the struct's own annotations.

### Discharge from Invariants

When `drop` calls `Box::from_raw(head.unwrap_some().as_ptr())`, the verifier needs `Allocated(raw, T, 1)` and `Owning(raw)`. Instead of requiring explicit annotations at the callsite, the verifier discharges these from the struct invariant:

```
#[rapx::invariant(Allocated(head.unwrap_some(), Node, 1))]
#[rapx::invariant(Owning(head.unwrap_some()))
```

The `discharge_from_field_invariant` mechanism traces the raw pointer back through up to 32 value-copy steps, finds the matching invariant on `LinkedList`, and discharges the obligation. This works for `Owning`, `Allocated`, `ValidPtr`, and `Typed` properties.

Two cross-kind implication rules from `primitive-sp.md` also apply:
- `Init ⇒ Typed`: `Init(field, T, count)` invariant discharges `Typed(field, T)`.
- `ValidPtr ⇒ Allocated ∧ InBound` (non-ZST): Discharges allocation and bounds checks.

## 8.4.7 Invless Mode

When struct invariants are not annotated, `invless` mode generates constructor-to-method sequences instead:

```
Constructor → Method
Constructor → Mutator → Method
```

The `linked_list_nonnull` example uses explicit invariants, so invless mode is not needed. Invless mode helps identify where invariants should be added: an unproved sequence indicates that a constructor's contracts are insufficient or a mutator invalidates a contract.

## 8.4.8 Raw Pointer Dereference

Direct `*ptr` and `*ptr = val` in unsafe blocks are treated as implicit checkpoints requiring `ValidPtr`, `Align`, and (for reads) `Typed`. The `linked_list_nonnull` example works entirely through safe abstractions (`NonNull`, `Option`, `Box`) and unsafe function calls (`as_mut`, `as_ref`, `from_raw`), so no raw pointer dereference checkpoints appear in its report.

## 8.4.9 Static Mut Access

`static mut` accesses generate implicit safety contracts requiring `ValidPtr`, `Align`, and (for reads) `Init`. The example does not use `static mut`, so no such checkpoints appear.

## 8.4.10 Auto-Repeat Detectors

Loop-carried properties (bounds checks, heap allocations) sometimes need more than one loop iteration to verify. The verifier starts at `postfix_repeat = 0` (each SCC body appears at most once). After the first round, dedicated auto-detectors run to decide whether deeper unrolling would help.

### Phase 1.5 — InBound

The InBound detector addresses **off-by-one errors**. At repeat 0, a bounds check inside a loop may appear `Proved` because the path only exercises the first iteration. A bug that triggers on the second iteration (e.g. `index == len`) would be missed.

- **Trigger**: all InBound results are `Proved` at the current depth.
- **Action**: progressively unroll up to 16 levels. At each level, re-run verification.
- **Stop conditions**: an InBound check becomes `Unproved` (found a bug — report it); or the `Proved` set stabilises (same as a previous level); or the `Proved` set oscillates between two states.

### Phase 1.6 — Deref / ValidPtr

Same logic as InBound, applied to `Deref` and `ValidPtr` results. These are compound properties (`Allocated ∧ InBound`) that depend on loop-carried allocation and bounds facts.

### Phase 1.7 — Allocated / Deref / ValidPtr

This detector handles **allocation reachability**. Fresh allocations inside loops (e.g. `Box::new` in a `for` body) produce facts (`PointsTo`, `KnownAllocated`) that the slicer needs to reach. The problem: at repeat 0 the allocation site appears on the path, but the fact may be too shallow in the trace for the forward verifier to carry it all the way to the callsite. Deeper unrolling repeats the loop body, extending the trace so the slicer can bridge the gap.

- **Trigger**: any Allocated / Deref / ValidPtr result is `Unproved` (excluding precision loss / connectivity failures).
- **Action**: retry with progressively deeper unrolling (up to 16). A new `VerifyDriver` is spawned at each level.
- **Stop conditions**: all targeted properties become `Proved`; or the set of still-unproved results converges to a previous level; or it oscillates.

In all three phases, earlier results from lower repeat levels are retained. Only newly `Proved` results from deeper levels are merged in — the mechanism is monotonic and never weakens a verdict.
