# Chapter 8.4. The Verification Pipeline

This chapter walks through the verification pipeline using the [`linked_list_nonnull`](https://github.com/safer-rust/RAPx/blob/main/rapx/tests/verify_cases/linked_list_nonnull/src/lib.rs) case study — a doubly-linked list built with `NonNull<Node>` pointers:

```rust
#[rapx::invariant(Align(head.unwrap_some(), Node))]
#[rapx::invariant(Allocated(head.unwrap_some(), Node, 1))]
#[rapx::invariant(Owning(head.unwrap_some()))]
struct LinkedList { head: Option<NonNull<Node>>, tail: Option<NonNull<Node>> }
```

The report shows two verification targets. These sections walk through the pipeline step by step, using the **first Ptr2Ref path** at bb10 as the single running example:

```
path [0, 1, 2, 3, 5, 7, 8, 9, 11, 14, 2, 3, 5, 7, 8, 9, 10]   →   Ptr2Ref | Proved
```

This path represents a two-element input: the first iteration (empty list, bb11 initialises head and tail), then the second iteration where `list.tail` is `Some` and `tail.as_mut()` is called at bb10.

## 8.4.1 Path Extraction

The first stage enumerates acyclic paths from function entry to each unsafe callsite using `PathGraph`'s SCC decomposition (see [§5.1 Path Analysis](./5.1-path.md) for the full algorithm). Each path is a sequence of basic block IDs.

The `for value in values { … }` loop contains an SCC with two branches (`tail` is `None` or `Some`). The path extractor unrolls this SCC once, producing paths for the first iteration (both branches) and subsequent iterations (each branch repeated). Paths are stored in a `PathTree` per callsite and capped at 1024. By default, verification starts at `repeat = 0` — each SCC body at most once. After the first round, **auto-detectors** (see [§8.4.6](#846-auto-repeat-detectors)) decide whether deeper unrolling is needed.

## 8.4.2 Backward Slicing

Take the path `[0, 1, 2, 3, 5, 7, 8, 9, 11, 14, 2, 3, 5, 7, 8, 9, 10]` — two-element input, first iteration initialises the empty list, second iteration calls `tail.as_mut()` at bb10. The relevant MIR blocks:

```
bb0:  {
    _2 = LinkedList { head: None, tail: None }   // empty list
}
bb1:  {
    _8 = <Vec<i32> as IntoIterator>::into_iter(move _1) → bb2
}
bb2:  {
    _11 = <IntoIter<i32> as Iterator>::next(&mut _8) → bb3   // for loop body
}
bb3:  {
    switchInt(discriminant(_11)) → [0: bb6, 1: bb5]   // None → exit, Some → process
}
bb5:  {
    _17 = Box::<Node>::new(move _18) → bb7             // heap allocate
}
bb7:  {
    _24 = move _17
    _23 = Box::<Node>::leak(move _24) → bb8            // transfer ownership
}
bb8:  {
    _22 = NonNull::<Node>::from(move _23) → bb9        // wrap in NonNull
}
bb9:  {
    _25 = discriminant((_2.1: Option<NonNull<Node>>))
    switchInt(move _25) → [0: bb11, 1: bb10]           // tail is None? Some?
}
bb10: {
    _30 = copy ((_2.1 as Some).0: NonNull<Node>)       // extract stored NonNull
    _34 = &mut _30
    _33 = NonNull::<Node>::as_mut(move _34) → bb12     // ← CHECKPOINT
}
bb11: {
    (_2.0) = Some(_27)                                  // set head
    (_2.1) = Some(_29)  // _29 = copy _22              // set tail
}
bb14: {
    goto → bb2                                          // loop back
}
```

The example path reaches bb10 **on the second iteration**: the first iteration went through bb11 (tail was `None`, initialised), then bb14 loops back, bb2-5-7-8 allocates a second node, bb9 dispatches again — this time tail is `Some`, so control flows to bb10.

The `BackwardSlicer` walks backward from `_33 = as_mut(move _34)`:

| Step | MIR | Action |
|------|-----|--------|
| bb10 | `_34 = &mut _30` | Reference to `_30` — follow its definition |
| bb10 | `_30 = copy ((_2.1 as Some).0)` | `(_2.1)` is `list.tail` — follow the struct field |
| bb11 | `(_2.1) = Some(_29), _29 = copy _22` | `_22` was defined at bb8 — include that chain |
| bb8 | `_22 = NonNull::from(move _23)` | Call effect `ReturnPointerFromArg` — include |
| bb7 | `_23 = Box::leak(move _24), _24 = move _17` | Follow `_17` to its definition — include both |
| bb5 | `_17 = Box::new(move _18)` | `Rvalue::ShallowInitBox` — allocation origin — **stop** |

The collected trace is: **bb5 → bb7 → bb8 → bb11 → (loop bb14) → bb10**. Rather than inlining the callee's MIR, the slicer recognises standard-library calls by their *effects* — `Box::new` provides a `BoxAllocation` effect (→ `KnownAllocated`), `NonNull::from` provides `ReturnPointerFromArg` (→ `PointsTo`). The second iteration's `Box::new` (the second pass through bb5) is not needed — the checkpoint at bb10 checks the *first* allocation's NonNull, which was created at the first pass through bb5. The slicer drops the second allocation chain and unrelated statements (iterator advance, value reads) because they don't affect `_30`'s provenance.

No `ContractFact` items are injected — `from_vec` has no `#[rapx::requires]` of its own.

## 8.4.3 Forward Verification

The `ForwardVerifier` walks forward along the same path, building abstract state:

**First pass through bb5** — `_17 = Box::<Node>::new(move _18)`

| Fact | Meaning |
|------|---------|
| `KnownNonZero(_17)` | Box pointer is non-null |
| `KnownAllocated(_17, Box<Node>, 1)` | `_17` points to a live heap allocation |

**First pass through bb7-8** — leak and wrap

`_23 = Box::leak(move _24)` transfers ownership (consumed, `&mut Node` returned). `_22 = NonNull::from(move _23)` applies `ReturnPointerFromArg` — the NonNull's internal pointer aliases the Box allocation. The verifier records `PointsTo(_22, _17)`.

**bb11** — store to struct

`_29 = copy _22` (value copy), then `(_2.1) = Some(_29)` writes to `list.tail`. The verifier records the assignment: `_2.1`'s `Some.0` field now contains `_22`.

**Loop back-edge (bb14 → bb2)** — the verifier resets path-local state (temporary locals are dead), but the struct field `(_2.1)` and its provenance facts are preserved across iterations. The second pass through bb5-8 creates a *second* allocation for the second node (with its own `KnownAllocated`), but these facts don't affect the bb10 checkpoint — bb10 checks the *first* allocation stored at bb11.

**bb9 → bb10** — the branch resolves `_25 = 1` (tail is `Some`). At `_30 = copy ((_2.1 as Some).0)`, the verifier resolves `_2.1` → `_22` → `_17` using the stored field value. `_34 = &mut _30` creates a reference — verifier records `PointsTo(_34, _17)` via the chain.

**bb10 checkpoint** — `_33 = as_mut(move _34)`. The `Ptr2Ref` checker resolves `_34 → _17`, queries `KnownNonZero(_17)` and `KnownAllocated(_17, Box<Node>, 1)`, and confirms both non-null and alignment (8 bytes for `Node`, satisfied by heap allocation). Result: `Ptr2Ref | Proved`.

## 8.4.4 SMT Check

Continuing the same path, the forward facts are encoded into SMT assertions. The `Ptr2Ref` checker (`ptr2ref.rs`) receives the checkpoint and resolves `_34 → _17`:

```
KnownNonZero(_17)                  →  _17 ≠ 0
KnownAllocated(_17, Box<Node>, 1)  →  alloc_begin(_17) ≠ null
```

The SMT obligation is `(_17 ≠ 0) ∧ (_17 % align_of(Node) == 0)`. The first conjunct falls out of `KnownNonZero`; the second is discharged by `KnownAllocated` (heap allocations are correctly aligned). Z3 confirms both trivially: `Ptr2Ref | Proved`.

For the struct invariant checkpoints at bb15, the same `PointsTo` chain resolves `head.unwrap_some()` → `_17`, and the same facts discharge `Align`, `Allocated`, and `Owning`. Each checker module (`align.rs`, `allocated.rs`, `owning.rs`) encodes its specific obligation from the shared fact set.

## 8.4.5 Struct Invariant Verification

Struct invariants are checked at every observable point. For this example, the report shows:

```
--- struct invariants ---
    checkpoint bb15:                    ← from_vec return
      path 0, 1, 2, 3, 6, 15:          ← loop never executes (empty vec)
        Align | Proved
        Allocated | Proved
        Owning | Proved
```

- `from_vec` is a constructor (returns `Self`): invariants are proved at `bb15` along all three reaching paths — the initial construction and two loop-body paths.
- `drop` is a method (`&mut self`): invariants are assumed at entry and checked at `bb7` (loop exit, where the list is empty) and `bb8` (after each `from_raw` call, where the head node has been freed and advanced).
- At `bb7`, the list is empty (`head` is `None`), so `head.unwrap_some()` vacuously satisfies `Align`, `Allocated`, and `Owning` since `Option::None` has no pointer to check.
- At `bb8`, invariants are re-checked after advancing the head pointer in the while loop. The check passes because the invariants track `Allocated` and `Owning` per-node, and the invariant discharge mechanism (below) proves these from the struct's own annotations.

### Discharge from Invariants

When `drop` calls `Box::from_raw(head.unwrap_some().as_ptr())`, the verifier needs `Allocated(raw, T, 1)` and `Owning(raw)`. Instead of requiring explicit annotations at the callsite, the verifier discharges these from the struct invariant:

```
#[rapx::invariant(Allocated(head.unwrap_some(), Node, 1))]
#[rapx::invariant(Owning(head.unwrap_some()))
```

The `discharge_from_field_invariant` mechanism traces the raw pointer back through up to 32 value-copy steps, finds the matching invariant on `LinkedList`, and discharges the obligation. This works for `Owning`, `Allocated`, `ValidPtr`, and `Typed` properties.

Two cross-kind implication rules from `primitive-sp.md` also apply:
- `Init ⇒ Typed`: `Init(field, T, count)` invariant discharges `Typed(field, T)`.
- `ValidPtr ⇒ Allocated ∧ InBound` (non-ZST): Discharges allocation and bounds checks.

## 8.4.6 Auto-Repeat Detectors

Loop-carried properties (bounds checks, heap allocations) sometimes need more than one loop iteration to verify. The verifier starts at `postfix_repeat = 0` (each SCC body appears at most once). After the first round, dedicated auto-detectors run to decide whether deeper unrolling would help.

### Phase 1.5 — InBound

The InBound detector addresses **off-by-one errors**. At repeat 0, a bounds check inside a loop may appear `Proved` because the path only exercises the first iteration. A bug that triggers on the second iteration (e.g. `index == len`) would be missed.

- **Trigger**: all InBound results are `Proved` at the current depth.
- **Action**: progressively unroll up to 16 levels. At each level, re-run verification.
- **Stop conditions**: an InBound check becomes `Unproved` (found a bug — report it); or the `Proved` set stabilises (same as a previous level); or the `Proved` set oscillates between two states.

### Phase 1.6 — Deref / ValidPtr

Same logic as InBound, applied to `Deref` and `ValidPtr` results. These are compound properties (`Allocated ∧ InBound`) that depend on loop-carried allocation and bounds facts.

### Phase 1.7 — Allocated / Deref / ValidPtr

This detector handles **allocation reachability**. Fresh allocations inside loops (e.g. `Box::new` in a `for` body) produce facts (`PointsTo`, `KnownAllocated`) that the slicer needs to reach. The problem: at repeat 0 the allocation site appears on the path, but the fact may be too shallow in the trace for the forward verifier to carry it all the way to the callsite. Deeper unrolling repeats the loop body, extending the trace so the slicer can bridge the gap.

- **Trigger**: any Allocated / Deref / ValidPtr result is `Unproved` (excluding precision loss / connectivity failures).
- **Action**: retry with progressively deeper unrolling (up to 16). A new `VerifyDriver` is spawned at each level.
- **Stop conditions**: all targeted properties become `Proved`; or the set of still-unproved results converges to a previous level; or it oscillates.

In all three phases, earlier results from lower repeat levels are retained. Only newly `Proved` results from deeper levels are merged in — the mechanism is monotonic and never weakens a verdict.
