# Chapter 6.3. Performance Bug Detection

This module identifies performance bottlenecks and inefficiencies using static analysis methods. After analysis, RAPx reports potential code inefficiencies and their positions. The implementation lives at `rapx/src/check/opt/`.

## Modules

The `opt/` module is organized into five categories of performance checks:

| Category | Directory | Focus |
|----------|-----------|-------|
| Bounds Checking | `checking/` | Unnecessary bounds checks |
| Encoding Checks | `checking/encoding_checking.rs` | Inefficient encoding patterns |
| Memory Cloning | `memory_cloning/` | Unnecessary clone operations |
| Data Collection | `data_collection/` | Suboptimal collection usage |
| Iterator | `iterator/` | Iterator-related inefficiencies |

## Bounds Checking

Rust performs automatic bounds checking on indexed accesses for safety. In performance-critical loops, these checks can be eliminated when the index is provably within bounds.

```rust
fn foo(mut a: Vec<i32>) {
    for i in 0..a.len() {
        a[i] = a[i] + 1;  // bounds check on every iteration
    }
}
```

RAPx detects such patterns and suggests replacing safe index accesses with unsafe APIs when the bounds invariant can be statically verified.

The bounds checking sub-modules include:

- **`bounds_extend.rs`**: Detects cases where `Vec::extend` or `Vec::extend_from_slice` would be more efficient than element-by-element push.
- **`bounds_len.rs`**: Analyzes index-vs-length relationships to prove bounds checks are redundant.
- **`bounds_loop_push.rs`**: Detects patterns where `for` loop iteration combined with `push` can be replaced with bulk operations.

## Memory Cloning

RAPx detects unnecessary clone operations where a cloned value is only used immutably. A borrow would suffice instead.

```rust
fn foo(a: &Vec<String>) {
    let mut b = HashSet::new();
    for i in a {
        let c = i.clone();   // unnecessary clone
        b.insert(c);
    }
}
```

The memory cloning checks include:

- **`used_as_immutable.rs`**: Detects cloned values that are only used in immutable contexts (e.g., as `HashSet` keys, `HashMap` keys, or read-only function arguments).
- **`hash_key_cloning.rs`**: Specifically targets clones used as keys in hash-based collections where the original borrowed value could be used instead.

**Note**: Developers need to manually verify whether removing the clone is semantically safe, as clone removal may change ownership semantics.

## Data Collection Optimization

RAPx detects suboptimal collection choices and suggests alternatives with better algorithmic complexity.

### Suboptimal Collection Type

```rust
fn foo(a: Vec<i32>, b: Vec<i32>) {
    for i in b.iter() {
        if a.contains(i) {}  // O(n) lookup on Vec, O(1) on HashSet
    }
}
```

RAPx suggests replacing `Vec` with `HashSet` when the dominating operation is membership testing.

The data collection checks include:

- **`initialization/`**: Detects patterns where collections can be pre-allocated or initialized more efficiently.
  - `local_set.rs`: Suggests `HashSet` initialization patterns.
  - `vec_init.rs`: Detects `Vec` initialization that can be optimized with `vec![]` or `with_capacity`.
- **`reallocation/`**: Detects patterns causing unnecessary reallocations.
  - `flatten_collect.rs`: Suggests `flatten().collect()` instead of nested iteration.
  - `unreserved_hash.rs`: Detects `HashMap`/`HashSet` usage without `reserve`.
  - `unreserved_vec.rs`: Detects `Vec` usage without `reserve` before known-size population.
- **`suboptimal/`**: Detects suboptimal algorithm patterns.
  - `participant.rs`: Detects repeated collection traversal patterns.
  - `slice_contains.rs`: Detects `Vec::contains` on large vectors where `HashSet` would be faster.
  - `vec_remove.rs`: Detects `Vec::remove` in loops (O(n²) when `swap_remove` would be O(n)).

### Iterator Optimization

```rust
fn foo(data: &[i32]) -> Vec<i32> {
    let mut result = Vec::new();
    for &item in data {
        result.push(item * 2);
    }
    result
}
```

RAPx suggests using iterator combinators (`map`, `collect`) which can be more efficient:

```rust
fn foo(data: &[i32]) -> Vec<i32> {
    data.iter().map(|&item| item * 2).collect()
}
```

The iterator check (`next_iterator.rs`) detects manual `for` loops that can be expressed as iterator chains, enabling the compiler to apply loop fusion and other optimizations.

## Encoding Checks

RAPx detects inefficient encoding patterns in string and byte manipulation:

- **`array_encoding.rs`**: Detects inefficient array-to-slice encoding patterns.
- **`string_lowercase.rs`**: Detects manual case conversion patterns that should use `to_lowercase()` or `to_ascii_lowercase()`.
- **`string_push.rs`**: Detects repeated `String::push` in loops that could use `push_str`, `write!`, or `extend`.
- **`vec_encoding.rs`**: Detects inefficient vector encoding patterns like repeated `push` when bulk operations are available.

## Usage

To detect performance bugs:

```shell
cargo rapx check -o
```

RAPx outputs warnings with file locations and suggested improvements. Each suggestion includes the source span for the detected pattern and a recommendation for the more efficient alternative.
