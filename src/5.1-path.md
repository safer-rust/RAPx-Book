# Chapter 5.1. Path Analysis

Path analysis extracts finite, acyclic execution paths from a function's control-flow graph (CFG). It is a foundational module — the alias analysis (Chapter 5.2), SafeDrop dangling pointer detection (Chapter 6.1), range analysis (Chapter 5.7), and the verification pipeline (Chapter 7) all depend on the path enumeration and reachability checking infrastructure provided by this module. The implementation lives at `rapx/src/analysis/path_analysis/`.

## Overview

Given a MIR function body, path analysis answers two questions:

1. **What paths exist?** — enumerate all structurally possible CFG paths from function entry to each exit point (or to each callsite), handling loops via strongly connected component (SCC) decomposition.
2. **Is a given path reachable?** — filter out paths that are structurally impossible due to contradictory branch constraints (e.g., the same enum discriminant cannot be both `First` and `Second` in a single execution).

The module consists of three main components: `PathGraph` (graph construction and reachability checking), `PathEnumerator` (path enumeration with a local SCC-path cache), and `PathAnalyzer` (top-level orchestrator that caches both graphs and path trees). `PathGraph` wraps a `ControlFlowGraph` and adds path-specific metadata (assigned locals per block, discriminant-to-source mappings) to enable constraint-based filtering.

## Quick Usage

```shell
# Default: postfix_repeat = 0
cargo rapx analyze paths

# With controlled SCC segment repetition
cargo rapx analyze paths --allow-pathseg-repeat 1
cargo rapx analyze paths --allow-pathseg-repeat 2
```

In code:

```rust
use analysis::path_analysis::default::PathAnalyzer;

let mut analyzer = PathAnalyzer::new(tcx, false);
analyzer.run();                          // postfix_repeat = 0
// or: analyzer.run_with_repeat(1);      // postfix_repeat = 1
let all_paths = analyzer.get_all_paths(); // FxHashMap<DefId, PathTree>
```

To check if a specific path is reachable:

```rust
let reachable = analyzer.check_path_reachability(def_id, &[0, 1, 3, 5]);
```

## PathTree: Prefix-Tree Path Storage

Paths are stored compactly using a `PathTree`, a prefix-tree (trie) that shares common prefixes across paths. This avoids duplicating repeated block sequences that appear in many paths and enables efficient prefix-based traversal.

```rust
pub struct PathTree {
    root: Option<PathNode>,
    len: usize,
}

pub struct PathNode {
    pub block: usize,
    pub children: Vec<PathNode>,
    pub is_path_end: bool,
}
```

`PathTree` supports:
- **`insert(path)`**: Insert a new path; returns `true` if newly added.
- **`contains(path)`**: Check if a complete path exists in the tree.
- **`iter()`**: Enumerate all paths as owned `Vec<usize>` sequences.
- **`walk_prefixes(target_block, f)`**: Walk the tree and call `f` with each unique prefix that ends at `target_block`. This stops at the target block without recursing into its children — useful for verification when we need paths up to a callsite.

No type aliases are used — `PathTree` and `FxHashMap<DefId, PathTree>` are used directly for clarity. The `PathMapWrapper` newtype provides `Display` formatting for the full map of per-function path trees:

```rust
pub struct PathMapWrapper(pub FxHashMap<DefId, PathTree>);
```

## PathExtractor (for verification)

The verification module (Chapter 7) uses a dedicated path extractor at `rapx/src/verify/path.rs`. `PathExtractor` builds on `PathGraph` to find paths reaching specific unsafe callsites:

```rust
let path_info = PathExtractor::new(tcx, def_id, callsites, allow_repeat).run();
let paths = path_info.paths_for(callsite.location());
```

It filters `PathEnumerator::enumerate_paths_repeat()` results to those containing the target block, truncates at the target, and deduplicates by prefix. Each `Path` is a sequence of `PathStep::Block` steps ending with a `PathStep::Callsite`.

## SCC-Aware Path Enumeration

Loops introduce strongly connected components (SCCs) that make naive path enumeration unbounded. `PathGraph` addresses this with an SCC tree decomposition and a controlled repetition strategy.

### SCC Tree Decomposition

Every loop in Rust MIR is a *natural loop* — it has a single **dominator** (entry block) through which all paths into the loop must pass. This property enables constructing a hierarchical **SCC tree**:

1. **SCC Detection**: `PathGraph::find_scc()` delegates to `ControlFlowGraph::find_scc()`, computing SCC nodes, the dominator (enter block), exit blocks, and child SCCs. Child SCC relationships are eagerly populated during `find_scc()` so that enumeration can be performed with only an immutable reference to the graph.
2. **SCC Tree Construction**: Starting from the outermost SCC, `populate_child_sccs()` identifies nested SCCs whose enter block lies within a parent SCC. This yields a tree where the root is the outermost SCC, children are nested sub-SCCs, and further nesting continues recursively.
3. **Top-Down Traversal**: The analysis starts from the outermost SCC and proceeds inward. When it reaches an inner SCC, it propagates the accumulated path constraints from the outer context to determine which inner paths are reachable. This top-down approach avoids the over-approximation of iterative bottom-up methods that ignore outer context constraints.

For example, given a CFG with a large SCC `{1, 22, 23, 2, 4, ..., 15}` whose dominator is block `1`, removing block `1` and starting from its successor `22` reveals a sub-SCC `{7, 8, 9, ..., 13}` with dominator `7`. These two dominators form a parent-child relationship in the SCC tree.

### Path Enumeration Through SCCs

`PathEnumerator::find_scc_paths()` and `PathEnumerator::find_scc_paths_repeat()` enumerate structurally possible paths through an SCC, treating child SCCs as atomic building blocks. The algorithm:

1. Recursively descends into child SCCs to pre-enumerate their internal paths.
2. Splices child SCC paths as atomic segments into the parent path.
3. Uses a `check_postfix_segment` mechanism: when a path returns to the SCC enter, the segment since the previous visit is extracted. If the same segment has been seen more than `postfix_repeat + 1` times, the branch is pruned — preventing unbounded unrolling while allowing controlled repetition.
4. Results are cached per `(def_id, scc_enter, postfix_repeat)` in the `PathEnumerator`'s instance-local cache to avoid recomputation.

A node sequence is worth further exploration only if the most recent traversal between two occurrences of the same dominator introduces at least one previously unvisited node. For instance, if the first traversal covers `7 → 8 → 10`, a second traversal covering `7 → 9 → 10` is valuable because it introduces block `9` and may lead to new analysis results. A traversal that merely repeats the same sequence without new blocks is pruned.

The `SccEnumeratedPath` struct records both the sequence of blocks within the SCC and the exit successors outside it:

```rust
pub struct SccEnumeratedPath {
    pub blocks: Vec<usize>,
    pub exit_successors: Vec<usize>,
}
```

### Postfix Repeat Control

The `postfix_repeat` parameter (exposed as `--allow-pathseg-repeat` in both the `analyze paths` and `verify` CLI commands) controls how many extra times the same SCC postfix segment may repeat:

- **`postfix_repeat = 0`** (default): Each distinct segment (block sequence from the SCC enter back to itself) appears at most once. This produces the minimum set of structurally distinct paths.
- **`postfix_repeat = 1`**: Each segment may repeat once (two total occurrences). Useful when alternating branch choices within a loop body produce different state transitions on successive iterations.
- **`postfix_repeat = N`**: Allows `N` extra repetitions, increasing path coverage at the cost of enumeration size.

The `check_postfix_segment` function in `graph.rs` extracts the segment between the previous and current occurrence of the SCC enter, maintains a per-segment count, and prunes when `count > postfix_repeat + 1`. This mechanism ensures that path enumeration remains finite while supporting analyses that require multi-iteration loop coverage (e.g., tracking loop-carried state propagation in alias analysis).

### Whole-CFG Paths

`PathEnumerator::enumerate_paths()` (and `PathEnumerator::enumerate_paths_repeat()` for repeated mode) assembles SCC paths into whole-CFG paths stored in a `PathTree`. Non-SCC blocks are traversed linearly; SCC regions are flattened into the enumerated set. Path enumeration is capped at `WHOLE_CFG_PATH_LIMIT` (4000) with a depth limit of 256.

## Path Reachability Filtering

Not every structurally possible path is actually reachable — branch conditions on enum discriminants can make certain combinations impossible. `is_path_reachable()` in `PathGraph` performs constraint-based filtering:

### Discriminant Tracking

For each `SwitchInt` terminator, the checker tracks the discriminant local. It maintains a `constraints: FxHashMap<usize, usize>` mapping discriminant locals to their known concrete values.

### Constraint Invalidation

When a block assigns to a local that appears in the constraints map, the constraint is removed — subsequent branches on that local are unconstrained until a new concrete value is observed.

### Switch Transition Validation

For each edge `cur → next` in the path:

1. If `cur` ends with `SwitchInt`, determine which target corresponds to `next`.
2. If the discriminant is a compile-time constant, check that `next` matches the resolved target.
3. If the discriminant has a known constraint value, check that `next` matches.
4. If the discriminant is unconstrained, record the new constraint from the taken branch so downstream branches can be validated.

### Otherwise Branch Inference

For `SwitchInt` with an "otherwise" branch, `infer_otherwise_value()` checks whether the explicit targets exhaust all possible values for the discriminant's type (e.g., `bool` has only 0 and 1; an enum has `N` variants). If exactly one value remains, the otherwise branch implies that value.

### Return Value

`is_path_reachable()` returns `true` if the path passes all checks, or `false` if any transition is provably contradictory. An extended variant, `is_path_reachable_with_constraints()`, also returns the reconstructed constraint map for downstream analyses.

### Worked Example: Correlated SwitchInt Branches

Consider a function with two `match` statements on the same enum discriminant:

```rust
enum Selector { First, Second }

fn foo<'a>(x: &'a i32, y: &'a i32, choice: Selector) -> &'a i32 {
    let a = match choice {
        Selector::First => x,
        Selector::Second => y,
    };
    match choice {
        Selector::First => a,
        Selector::Second => x,
    }
}
```

The corresponding MIR contains two `SwitchInt` terminators on the same discriminant `_3`:

```
bb0:  _5 = discriminant(_3)
      switchInt(_5) -> [0: bb3, 1: bb1, otherwise: bb2]
bb1:  _4 = _2; goto bb4       // _5 = 1 (Second → a = y)
bb3:  _4 = _1; goto bb4       // _5 = 0 (First → a = x)
bb4:  _6 = discriminant(_3)
      switchInt(_6) -> [0: bb6, 1: bb5, otherwise: bb2]
bb5:  _0 = _1; goto bb7       // _6 = 1 → return x
bb6:  _0 = _4; goto bb7       // _6 = 0 → return a
```

At the first `SwitchInt`, `_5` is unconstrained — both branches are tracked, recording `_5 = 0` for `bb3` and `_5 = 1` for `bb1`. The `discriminants` map (`_5 → _3`) links the temp back to `_3`.

At the second `SwitchInt` in `bb4`, the discriminant local is `_6`, also linked to `_3` via `discriminants`. The checker sees the existing constraint on `_3` from the first branch and validates consistency:

- **Path: bb3 (choice=First) → bb4 → bb6 (choice=First)**: `_3 = 0` matches `switchInt → bb6` (value 0). Reachable.
- **Path: bb3 (choice=First) → bb4 → bb5 (choice=Second)**: `_3 = 0` contradicts `switchInt → bb5` (value 1). Pruned.
- **Path: bb1 (choice=Second) → bb4 → bb5 (choice=Second)**: `_3 = 1` matches `switchInt → bb5` (value 1). Reachable.
- **Path: bb1 (choice=Second) → bb4 → bb6 (choice=First)**: `_3 = 1` contradicts `switchInt → bb6` (value 0). Pruned.

This filtering reduces the path count from 4 to 2, eliminating the impossible combination where `choice` is simultaneously `First` and `Second`. Without it, the analysis would incorrectly conclude that the return value may alias `y`, when in fact only `x` is possible on any real execution.

### Constraint Propagation Across SCCs

The `check_segment_reachability_with()` method in `PathGraph` extends the constraint-based filtering to SCC boundaries. During top-down SCC traversal, it propagates accumulated discriminant constraints from outer SCCs into nested SCCs, ensuring that inner SCC path enumeration only explores paths consistent with the outer context. For nested loops with immutable guard variables (e.g., a `choice` parameter that drives branch decisions in both an outer and inner loop), this prevents exploring unreachable combinations where the inner loop's branch conflicts with the outer loop's constraint.

When an `enum` discriminant is constrained to a specific variant before entering a loop, the constraint persists across all loop iterations (since the immutable variable cannot be reassigned). The `check_segment_reachability_with()` method uses the `discriminants` mapping to propagate the constraint through temporary discriminant-read locals, and `is_path_reachable_with_constraints()` computes per-path constraint summaries for use in downstream analyses.

## Data Structures

### PathGraph

```rust
pub struct PathGraph<'tcx> {
    pub cfg: ControlFlowGraph<'tcx>,
    pub assigned_locals: Vec<FxHashSet<usize>>,
    pub discriminants: FxHashMap<usize, usize>,
}
```

- `cfg`: The standard CFG built from MIR basic blocks.
- `assigned_locals`: Per-block set of MIR locals that are assigned (written to) in that block. Used for constraint invalidation.
- `discriminants`: Maps a discriminant-reading local back to the source enum local (e.g., `_5 = discriminant(_3)` produces `_5 → _3`). This enables tracking enum values even when the discriminant is extracted into a temporary.

`PathGraph` is responsible for graph construction (`new()`) and reachability checking (`is_path_reachable()`, `check_segment_reachability_with()`). Path enumeration has been extracted to `PathEnumerator`.

### PathEnumerator

```rust
pub struct PathEnumerator<'g, 'tcx> {
    graph: &'g PathGraph<'tcx>,
    scc_path_cache: FxHashMap<(DefId, usize, usize), Vec<SccEnumeratedPath>>,
}
```

`PathEnumerator` takes an immutable reference to a `PathGraph` (which must have `find_scc()` called beforehand) and enumerates paths into a `PathTree`. It owns an instance-local SCC-path cache, replacing the previous thread-local global cache. Key methods:

- **`enumerate_paths()`**: Enumerate all whole-CFG paths (equivalent to `enumerate_paths_repeat(0)`).
- **`enumerate_paths_repeat(postfix_repeat)`**: Enumerate paths with controlled SCC segment repetition.
- **`find_scc_paths()` / `find_scc_paths_repeat()`**: Enumerate paths through a single SCC, recursing into child SCCs.

### PathAnalyzer

```rust
pub struct PathAnalyzer<'tcx> {
    pub tcx: TyCtxt<'tcx>,
    pub debug: bool,
    pub paths: FxHashMap<DefId, PathTree>,
    pub graphs: FxHashMap<DefId, PathGraph<'tcx>>,
}
```

`PathAnalyzer` is the top-level analysis entry point. It orchestrates per-function path enumeration via `start_path_analysis_for_defid()` and `start_path_analysis_for_defid_with_repeat()`, which construct a `PathGraph`, call `find_scc()`, create a `PathEnumerator` to produce a `PathTree`, and cache both the graph and path tree. The `run_with_repeat(postfix_repeat)` method scans all functions; `check_path_reachability()` reuses the cached `PathGraph` rather than constructing a fresh one, avoiding redundant MIR parsing and CFG construction.

## Relationship to Other Modules

- **Alias Analysis (Chapter 5.2)**: The MOP-based `AliasGraph` wraps a `PathGraph` for path-sensitive alias checking. The SCC decomposition, top-down traversal, and path reachability checks are all shared with the path analysis module. `AliasGraph::enumerate_paths()` creates a `PathEnumerator` internally. The `check_segment_reachability_with()` method propagates discriminant constraints across SCC boundaries during alias traversal.
- **SafeDrop (Chapter 6.1)**: Reuses `PathGraph` for path-sensitive dangling pointer detection. Per-block alias facts (`AliasBlockFacts`) and path-sensitive alias state are tracked along enumerated paths.
- **Verification (Chapter 7)**: `PathExtractor` in `verify/path.rs` wraps `PathGraph` and uses `PathEnumerator` to find paths reaching specific unsafe callsites. The `allow_pathseg_repeat` parameter controls how many extra SCC postfix repetitions are permitted.
- **Range Analysis (Chapter 5.7)**: `PathGraph` is reused for path-constraint extraction in `RangeAnalyzer`'s `start_path_constraints_analysis()` method.

## Test Examples

Dedicated path analysis tests live at `rapx/tests/paths/`. Each test exercises a specific SCC enumeration scenario and asserts both exact path counts and specific block-index sequences.

### Single SCC: `scc_simple`

Tests single-loop SCC decomposition with conditional branches inside the loop body. The function `classify` (11 paths at `postfix_repeat=0`, 39 at repeat=1, 128 at repeat=2) shows how path count grows with controlled repetition:

```rust
fn classify(x: i32) -> i32 {
    let mut i = 0;
    loop {
        if i >= x { break; }
        if i % 2 == 0 { i += 1; } else { i += 2; }
    }
    i
}
```

The test verifies that the zero-iteration path `[0, 1, 2]`, the even-branch path `[0, 1, 3, 4, 5, 6, 7, 10, 1, 2]`, the odd-branch path `[0, 1, 3, 4, 5, 8, 9, 10, 1, 2]`, and cross-branch combos are all enumerated.

### Nested SCC: `scc_nested`

Tests nested-loop SCC decomposition where an inner `loop` lives inside an outer `loop`. The function `walk` (11 paths at repeat=0) demonstrates child SCC paths spliced as atomic segments into the parent SCC:

```rust
fn walk(rows: i32, cols: i32) -> i32 {
    let mut total = 0;
    let mut r = 0;
    loop {
        if r >= rows { break; }
        let mut c = 0;
        loop {
            if c >= cols { break; }
            total += 1; c += 1;
        }
        r += 1;
    }
    total
}
```

The test verifies paths like `[0, 1, 2]` (zero iterations), `[0, 1, 3, 4, 6, 7, 8, 4, 5, 9, 1, 2]` (one outer iteration with one inner iteration), and multi-iteration variants with inner loop repetition.

### Nested SCC with `#[rapx::verify]`: `scc1`

The `scc1` test uses `#[rapx::verify]` annotated functions with single and nested SCCs (alignment check retry loops). `read1` produces 4 SCC paths; `read2` (nested) produces 10. The test validates paths like `[0, 1, 2, 3, 4, 8, 9]` (direct success) and `[0, 1, 2, 3, 5, 6, 1, 2, 3, 4, 8, 9]` (retry → success).

### Branch-based paths: `if_else`

Tests path enumeration through `if`/`else` branches (no SCC). `read1` produces 6 paths covering both alignment branches; `read2` produces 3 paths through `ManuallyDrop`. Each test asserts the full set of expected block-index sequences.

### Repeat tests: `scc_repeat1` / `scc_repeat2`

The `scc_repeat1` and `scc_repeat2` tests run `--allow-pathseg-repeat 1` and `2` against `scc_simple`, verifying that path counts increase predictably: `classify` goes from 11 → 39 → 128 paths, and `early_exit` from 3 → 4 → 5 paths. These tests confirm that the `postfix_repeat` parameter produces the expected cardinality growth while maintaining finite enumeration.
