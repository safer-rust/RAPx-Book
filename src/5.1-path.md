# Chapter 5.1. Path Analysis

Path analysis extracts finite, acyclic execution paths from a function's control-flow graph (CFG). It is a foundational module — the alias analysis (Chapter 5.2) and the verification pipeline (Chapter 7) both depend on path enumeration and reachability checking provided by this module. The implementation lives at `rapx/src/analysis/path_analysis/`.

## Overview

Given a MIR function body, path analysis answers two questions:

1. **What paths exist?** — enumerate all structurally possible CFG paths from function entry to each exit point (or to each callsite), handling loops via strongly connected component (SCC) decomposition.
2. **Is a given path reachable?** — filter out paths that are structurally impossible due to contradictory branch constraints (e.g., the same enum discriminant cannot be both `First` and `Second` in a single execution).

The `PathGraph` structure in `graph.rs` is the core data structure. It wraps a `ControlFlowGraph` and adds path-specific metadata (assigned locals per block, discriminant-to-source mappings) to enable constraint-based filtering.

## Quick Usage

```shell
cargo rapx analyze paths
```

In code:

```rust
use analysis::path_analysis::default::PathAnalyzer;

let mut analyzer = PathAnalyzer::new(tcx, false);
analyzer.run();
let all_paths = analyzer.get_all_paths(); // PathMap
```

## PathExtractor (for verification)

The verification module (Chapter 7) uses a dedicated path extractor at `rapx/src/verify/path.rs`. `PathExtractor` builds on `PathGraph` to find paths reaching specific unsafe callsites:

```rust
let path_info = PathExtractor::new(tcx, def_id, callsites, allow_repeat).run();
let paths = path_info.paths_for(callsite.location());
```

It filters `PathGraph::enumerate_paths_repeat()` results to those containing the target block, truncates at the target, and deduplicates by prefix.

## SCC-Aware Path Enumeration

Loops introduce strongly connected components (SCCs) that make naive path enumeration unbounded. `PathGraph` addresses this with an SCC tree decomposition and a controlled repetition strategy.

### SCC Tree Decomposition

Every loop in Rust MIR is a *natural loop* — it has a single **dominator** (entry block) through which all paths into the loop must pass. This property enables constructing a hierarchical **SCC tree**:

1. **SCC Detection**: `find_scc()` delegates to `ControlFlowGraph::find_scc()`, computing SCC nodes, the dominator (enter block), exit blocks, and child SCCs.
2. **SCC Tree Construction**: Starting from the outermost SCC, `populate_child_sccs()` identifies nested SCCs whose enter block lies within a parent SCC. This yields a tree where the root is the outermost SCC, children are nested sub-SCCs, and further nesting continues recursively.
3. **Top-Down Traversal**: The analysis starts from the outermost SCC and proceeds inward. When it reaches an inner SCC, it propagates the accumulated path constraints from the outer context to determine which inner paths are reachable. This top-down approach avoids the over-approximation of iterative bottom-up methods that ignore outer context constraints.

For example, given a CFG with a large SCC `{1, 22, 23, 2, 4, ..., 15}` whose dominator is block `1`, removing block `1` and starting from its successor `22` reveals a sub-SCC `{7, 8, 9, ..., 13}` with dominator `7`. These two dominators form a parent-child relationship in the SCC tree.

### Path Enumeration Through SCCs

`find_scc_paths()` and `find_scc_paths_repeat()` enumerate structurally possible paths through an SCC, treating child SCCs as atomic building blocks. The algorithm:

1. Recursively descends into child SCCs to pre-enumerate their internal paths.
2. Splices child SCC paths as atomic segments into the parent path.
3. Uses a `check_postfix_segment` mechanism: when a path returns to the SCC enter, the segment since the previous visit is extracted. If the same segment has been seen more than `postfix_repeat + 1` times, the branch is pruned — preventing unbounded unrolling while allowing controlled repetition.
4. Results are cached per `(def_id, scc_enter, postfix_repeat)` to avoid recomputation.

A node sequence is worth further exploration only if the most recent traversal between two occurrences of the same dominator introduces at least one previously unvisited node. For instance, if the first traversal covers `7 → 8 → 10`, a second traversal covering `7 → 9 → 10` is valuable because it introduces block `9` and may lead to new analysis results. A traversal that merely repeats the same sequence without new blocks is pruned.

The `SccEnumeratedPath` struct records both the sequence of blocks within the SCC and the exit successors outside it:

```rust
pub struct SccEnumeratedPath {
    pub blocks: Vec<usize>,
    pub exit_successors: Vec<usize>,
}
```

### Postfix Repeat Control

The `postfix_repeat` parameter (exposed as `--allow-pathseg-repeat` in the verification CLI) controls how many extra times the same SCC postfix segment may repeat:

- **`postfix_repeat = 0`** (default): Each distinct segment (block sequence from the SCC enter back to itself) appears at most once. This produces the minimum set of structurally distinct paths.
- **`postfix_repeat = 1`**: Each segment may repeat once (two total occurrences). Useful when alternating branch choices within a loop body produce different state transitions on successive iterations.
- **`postfix_repeat = N`**: Allows `N` extra repetitions, increasing path coverage at the cost of enumeration size.

The `check_postfix_segment` function in `graph.rs` extracts the segment between the previous and current occurrence of the SCC enter, maintains a per-segment count, and prunes when `count > postfix_repeat + 1`. This mechanism ensures that path enumeration remains finite while supporting analyses that require multi-iteration loop coverage (e.g., tracking loop-carried state propagation in alias analysis).

### Whole-CFG Paths

`enumerate_paths_repeat()` assembles SCC paths into whole-CFG paths. Non-SCC blocks are traversed linearly; SCC regions are flattened into the enumerated set. The result is a `Vec<Vec<usize>>` of block-index sequences, capped at `WHOLE_CFG_PATH_LIMIT` (4000) with a depth limit of 256.

## Path Reachability Filtering

Not every structurally possible path is actually reachable — branch conditions on enum discriminants can make certain combinations impossible. `is_path_reachable()` in `PathGraph` performs constraint-based filtering:

### Discriminant Tracking

For each `SwitchInt` terminator, the checker tracks the discriminant local. It maintains a `constraints: FxHashMap<usize, usize>` mapping discriminant locals to their known concrete values.

### Constraint Invalidation

When a block assigns to a local that appears in the constraints map, the constraint is removed — subsequent branches on that local are unconstrained until a new concrete value is observed.

### Switch Transition Validation

For each edge `cur → next` in the path:

1. If `cur` ends with `SwitchInt`, determine which target corresponds to `next`.
2. If the discriminant is a compile-time constant, check that `next` matches the resolved target.
3. If the discriminant has a known constraint value, check that `next` matches.
4. If the discriminant is unconstrained, record the new constraint from the taken branch so downstream branches can be validated.

### Otherwise Branch Inference

For `SwitchInt` with an "otherwise" branch, `infer_otherwise_value()` checks whether the explicit targets exhaust all possible values for the discriminant's type (e.g., `bool` has only 0 and 1; an enum has `N` variants). If exactly one value remains, the otherwise branch implies that value.

### Return Value

`is_path_reachable()` returns `true` if the path passes all checks, or `false` if any transition is provably contradictory.

An extended variant, `is_path_reachable_with_constraints()`, also returns the reconstructed constraint map for downstream analyses.

### Worked Example: Correlated SwitchInt Branches

Consider a function with two `match` statements on the same enum discriminant:

```rust
enum Selector { First, Second }

fn foo<'a>(x: &'a i32, y: &'a i32, choice: Selector) -> &'a i32 {
    let a = match choice {
        Selector::First => x,
        Selector::Second => y,
    };
    match choice {
        Selector::First => a,
        Selector::Second => x,
    }
}
```

Without path reachability filtering, the analysis would see 4 structurally possible paths (2×2 branch combinations). However, `is_path_reachable()` prunes 2 of them:

The corresponding MIR contains two `SwitchInt` terminators on the same discriminant `_3`:

```
bb0:  _5 = discriminant(_3)
      switchInt(_5) -> [0: bb3, 1: bb1, otherwise: bb2]
bb1:  _4 = _2; goto bb4       // _5 = 1 (Second → a = y)
bb3:  _4 = _1; goto bb4       // _5 = 0 (First → a = x)
bb4:  _6 = discriminant(_3)
      switchInt(_6) -> [0: bb6, 1: bb5, otherwise: bb2]
bb5:  _0 = _1; goto bb7       // _6 = 1 → return x
bb6:  _0 = _4; goto bb7       // _6 = 0 → return a
```

At the first `SwitchInt`, `_5` is unconstrained — both branches are tracked, recording `_5 = 0` for `bb3` and `_5 = 1` for `bb1`. The `discriminants` map (`_5 → _3`) links the temp back to `_3`.

At the second `SwitchInt` in `bb4`, the discriminant local is `_6`, also linked to `_3` via `discriminants`. The checker sees the existing constraint on `_3` from the first branch and validates consistency:

- **Path: bb3 (choice=First) → bb4 → bb6 (choice=First)**: `_3 = 0` matches `switchInt → bb6` (value 0). Reachable.
- **Path: bb3 (choice=First) → bb4 → bb5 (choice=Second)**: `_3 = 0` contradicts `switchInt → bb5` (value 1). Pruned.
- **Path: bb1 (choice=Second) → bb4 → bb5 (choice=Second)**: `_3 = 1` matches `switchInt → bb5` (value 1). Reachable.
- **Path: bb1 (choice=Second) → bb4 → bb6 (choice=First)**: `_3 = 1` contradicts `switchInt → bb6` (value 0). Pruned.

This filtering reduces the path count from 4 to 2, eliminating the impossible combination where `choice` is simultaneously `First` and `Second`. Without it, the analysis would incorrectly conclude that the return value may alias `y`, when in fact only `x` is possible on any real execution.

## Data Structures

### PathGraph

```rust
pub struct PathGraph<'tcx> {
    pub cfg: ControlFlowGraph<'tcx>,
    pub assigned_locals: Vec<FxHashSet<usize>>,
    pub discriminants: FxHashMap<usize, usize>,
}
```

- `cfg`: The standard CFG built from MIR basic blocks.
- `assigned_locals`: Per-block set of MIR locals that are assigned (written to) in that block. Used for constraint invalidation.
- `discriminants`: Maps a discriminant-reading local back to the source enum local (e.g., `_5 = discriminant(_3)` produces `_5 → _3`). This enables tracking enum values even when the discriminant is extracted into a temporary.

### PathSet / PathMap

```rust
pub type PathSet = Vec<Vec<usize>>;
pub type PathMap = FxHashMap<DefId, PathSet>;
```

`PathMap` maps each function's `DefId` to its set of whole-CFG paths (list of block-index sequences).

## Test Example: Nested Loop SCC Decomposition

The test at `rapx/tests/range/range_1/src/main.rs` demonstrates how `PathGraph` handles nested loops:

```rust
fn main() {
    let mut k = 0;
    while k < 100 {           // outer loop — SCC with dominator at loop header
        let mut i = 0;
        let mut j = k;
        while i < j {         // inner loop — nested sub-SCC
            i += 1;
            j -= 1;
        }
        k += 1;
    }
}
```

The MIR for this function contains two nested natural loops. `PathGraph::find_scc()` decomposes them into an SCC tree:

- The **outer SCC** includes the `while k < 100` loop body and the increment `k += 1`. Its dominator is the block evaluating `k < 100`.
- The **inner SCC** (a child of the outer SCC) includes the `while i < j` body with `i += 1` and `j -= 1`. Its dominator is the block evaluating `i < j`.

When `enumerate_paths_repeat()` runs, each SCC is flattened. The outer SCC contains the inner SCC as a child, so its paths are spliced in as atomic segments. The resulting whole-CFG paths include both the loop body and the final exit path for each possible iteration count (bounded by the postfix repeat limit).
