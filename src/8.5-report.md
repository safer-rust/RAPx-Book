# Chapter 8.5. Report, Examples, and Limitations

## 8.5.1 Verification Report

The `VerificationReport` ([`report.rs`](https://github.com/safer-rust/RAPx/blob/main/rapx/src/verify/report.rs)) aggregates results for all (callsite, path, property) triples. Each `PropertyCheckResult` records the callsite location, path/property indices, the property and its result (`Proved` / `Unproved`), and forward/backward visit diagnostics.

### `--debug-contracts` Contract Debug Output

The `--debug-contracts` flag prints every resolved contract for each verification target and **exits without running verification** (like `--prepare-targets`). Targets with no meaningful contracts are skipped; each unique `PropertyKind` appears once per target, with caller annotations taking priority over callee contracts.

```
============================================================
[rapx::debug-contracts] Expanded Contract Assertions
============================================================

fn my_crate::unsafe_fn
----------------------------------------------------------------------------
  Safety Tag: ValidPtr(ptr, T, len)
    Meaning:  same_alloc([ptr, ptr + sizeof(T) * len])
  Safety Tag: Align(ptr, T)
    Meaning:  (ptr as usize) % align_of::<T>() == 0
  Safety Tag: InBound(self, index)
    Meaning:  0 <= index < self.len()
```

Each contract is displayed in a two-line format — a `Safety Tag:` line showing the annotation syntax, and an indented `Meaning:` line with the formal constraint. All 31 property kinds have descriptive ASCII meanings (see [Chapter 8.3](./8.3-contracts.md)).

The `emit_verify_summary` function in [`driver.rs`](https://github.com/safer-rust/RAPx/blob/main/rapx/src/verify/driver.rs) groups results by checkpoint (callsites, raw pointer dereferences, and static mut accesses) and prints a compact summary:

```
============================================================
[rapx::verify] function: my_crate::my_function
============================================================
  --- unsafe checkpoints ---
      unsafe checkpoint: bb3 -> std::ptr::read
        path [0, 1, 3]:
          ValidPtr | Proved
          Align | Proved
          Typed | Proved
  result: SOUND
```

If any property is unproved, the function is reported as `UNSOUND` with a count of unproved properties:

```
      unsafe checkpoint: bb5 -> std::ptr::read
        path [0, 2, 4, 5]:
          ValidPtr | Proved
          Align | Proved
          Typed | Unproved
  result: UNSOUND (1 unproved)
```

When struct invariants are verified, they appear in a separate section:

```
  --- struct invariants ---
      checkpoint bb6:
        path [0, 1, 2, 6]:
          ValidPtr | Proved
          Align | Proved
  result: SOUND
```

Hazard contracts (flagged with `kind = "hazard"` or `#[rapx::hazard]`) are printed with a `[hazard]` prefix and excluded from the unproved count:

```
      unsafe checkpoint: bb4 -> std::ptr::write
        path [0, 1, 4]:
          [hazard] ValidNum | Unproved
  result: SOUND
```

## 8.5.2 Interpreting the Report

| Result | Meaning | Action |
|--------|---------|--------|
| All `Proved` → `SOUND` | The verifier confirmed all contracts for all paths. | No action needed; the function is verified. |
| Some `Unproved` → `UNSOUND` | At least one property could not be proved for at least one path. | Inspect the unproved property and path. The verifier may lack facts (e.g., missing `assert!`, unknown call effect), or there may be genuine UB. |
| `CheckResult::Unknown` | The contract tag is not yet supported. | File an issue or contribute support for the contract tag. |

## 8.5.3 Forward/Backward Visit Diagnostics

When a property is `Unproved`, the report includes diagnostics showing which MIR items were collected and which facts were established:

```
        path [0, 2, 4, 5]:
          Typed | Unproved
          backward items: [
            Statement(_4 = ptr::read(move _3)),
            Statement(_3 = &raw const (*_2)),
            Statement(_2 = &(*_1).0),
            ContractFact(ValidPtr(self.0, T, 1)),
          ]
          forward facts: [
            PointsTo(_3, (*_2)),
            PointsTo(_2, (*_1).0),
          ]
```

Here the `Typed` check failed because the forward facts don't establish that `(*_1).0` holds a valid value of type `T`. The missing piece might be a `KnownInit` fact — supplied by adding `#[rapx::requires(Init(self.0, T, 1))]`, or by annotating a preceding `ptr::write` in the call summary database.

## 8.5.4 Example: Field Invariant Discharge in `Vec::into_raw_parts`

The `into_raw_parts` method reads a `Vec`'s internal pointer field through `ptr::read`, which requires `ValidPtr(ptr, T, 1)`. Instead of manually tracing every MIR statement, the verifier discharges this obligation from the struct field invariant:

```rust
#[rapx::invariant(ValidPtr(self.ptr, T, self.cap))]
pub struct Vec<T> {
    ptr: *mut T,
    len: usize,
    cap: usize,
}

impl<T> Vec<T> {
    pub fn into_raw_parts(self) -> (*mut T, usize, usize) {
        // ptr::read on self.ptr — requires ValidPtr(ptr, T, 1)
        let me = ManuallyDrop::new(self);
        unsafe { (ptr::read(&me.ptr), me.len, me.cap) }
    }
}
```

The verification proceeds as follows:

1. **Collection:** `build_type_invariants_from_params()` injects the `Vec`'s struct invariants as `caller_requires` — the verifier enters the function already knowing `ValidPtr(self.ptr, T, self.cap)`.
2. **Path Extraction:** Enumerate paths from function entry to the `ptr::read` callsite.
3. **SMT Check — Field Invariant Discharge:** When the checker needs `ValidPtr(ptr, T, 1)` at the `ptr::read` callsite, `discharge_from_field_invariant()` traces `ptr` back to `me.ptr` → `self.ptr`. It finds that `Vec<T>` declares `ValidPtr(self.ptr, T, self.cap)`. Since `self.cap >= 1` (proved from the `Vec` invariant), the obligation for 1 element is covered. `Align` follows from struct layout. Both are **Proved** — no forward/backward visit needed for these properties.

This is the same mechanism described in [Chapter 8.4.9](./8.4-pipeline.md#849-discharge-from-struct-field-invariants). It works for `Owning`, `Allocated`, `ValidPtr`, and `Typed` obligations on struct fields, using the struct's own invariants as discharge proofs.

## 8.5.5 Example: A Bug Across Branches

```rust
unsafe fn buggy_read(buffer: *const u8, len: usize, index: usize, cond: bool) -> u8 {
    if cond {
        if index < len {
            return *buffer.add(index);  // Safe path
        }
    }
    *buffer.add(index)  // Bug: no bounds check on this path
}
```

The verifier produces:

```
  --- unsafe checkpoints ---
      unsafe checkpoint: bb7 -> ptr::add
        path [0, 1, 2, 3, 5, 7]:          // cond=false → no bounds check
          ValidPtr | Unproved
          Align | Proved
          Typed | Proved
        path [0, 1, 2, 4, 7]:             // cond=true, index<len
          ValidPtr | Proved
          Align | Proved
          Typed | Proved
  result: UNSOUND (1 unproved)
```

On the `cond=false` path, no `ValidNum(index < len)` constraint is established, so `ValidPtr(buffer.add(index), u8, 1)` cannot be proved. Fix by adding the bounds check to all paths.

## 8.5.6 Current Limitations

- **Path enumeration is capped** at 1024 paths per callsite. Functions with very complex control flow may exceed this limit; the verifier processes the first 1024 paths and warns.
- **Postfix repeat convergence** may not exhaustively unroll deeply nested SCCs within a practical bound. Off-by-one bugs requiring more than 16 loop iterations may escape detection.
- **Inter-procedural analysis is limited** to pre-defined call summaries; full cross-function MIR-level verification is not supported. Custom unsafe functions calling other custom unsafe functions require explicit contracts on both or a manual call-summary entry.
- **The SMT encoding approximates** numeric constraints. Non-linear arithmetic (multiplication of two variables) is not well-supported and may yield `Unproved` even when the property holds.
- **Struct invariant verification** for method sequences exceeding the maximum chain depth is not explored.
- **Standard library contracts** are maintained manually in `std-public-contracts.json`: 330 functions, 723 contract entries, 31 property kinds. Functions not covered are silently skipped.
- **18 of the 31 `PropertyKind` variants** have dedicated SMT modules; the rest produce `CheckResult::Unknown` and do not block soundness. Unrecognized tags are classified as `Unknown` and silently skipped.
- **Concurrent code** (`Arc`, `Mutex`, atomics) is not modeled; the analysis assumes single-threaded execution.
- **Inline assembly** (`asm!`) is not analyzed; such functions are treated as having unknown effects.

## 8.5.7 Performance Characteristics

| Factor | Impact |
|--------|--------|
| Number of paths per callsite | Linear — each path is analyzed independently |
| Path length (MIR statements) | Linear in backward/forward visit time |
| Number of property kinds | Linear — each property checked independently |
| SMT encoding complexity | Dominated by `ValidNum` and `InBound` integer-arithmetic encodings |
| Solver timeout | Default 5 seconds per check; complex constraints may time out |

## 8.5.8 Future Work

- **Full inter-procedural verification**: Replace call summaries with MIR-level cross-function analysis.
- **Postcondition inference**: Automatically derive postconditions from function bodies.
- **Lifetime-aware pointer analysis**: Integrate borrow-checker information for more precise modeling.
- **Concurrency model**: Extend the abstract domain to model thread-local state and detect data races.
