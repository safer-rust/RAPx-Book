# Chapter 8.5. Verification Report

## 8.5.1 Report Format

When verification runs, each function is printed with its checkpoints (unsafe call sites), the paths through the control-flow graph to reach them, and the verdict for each property on each path. Properties are either `Proved`, `Unproved`, or `Unknown`.

At the end of each function, a `result:` line summarizes the overall verdict:
- `SOUND` — all properties proved on all paths.
- `UNSOUND (N unproved)` — at least one property could not be proved.

## 8.5.2 Example: Linked List with `NonNull`

The [`linked_list_nonnull`](https://github.com/safer-rust/RAPx/blob/main/rapx/tests/verify_cases/linked_list_nonnull/src/lib.rs) case study demonstrates struct invariants, ownership transfers, and hazard contracts on a doubly-linked list built with `NonNull<Node>` pointers. The struct declares:

```rust
#[rapx::invariant(Align(head.unwrap_some(), Node))]
#[rapx::invariant(Allocated(head.unwrap_some(), Node, 1))]
#[rapx::invariant(Owning(head.unwrap_some()))]
struct LinkedList { head: Option<NonNull<Node>>, tail: Option<NonNull<Node>> }
```

Key points in this report:

- **from_vec**: `Ptr2Ref` is satisfied at both `NonNull::as_mut` call sites because the `NonNull` was just created from a `Box::leak`. The struct invariants are checked at the return point (`bb15`) along all three reaching paths.
- **drop**: The loop body's first iteration is a single path; subsequent iterations follow the longer path. `Allocated` and `Owning` are proved at `Box::from_raw` because the invariant supplies `Allocated(head.unwrap_some(), Node, 1)` and `Owning(head.unwrap_some())`. `Alias` is shown with `[hazard]` — even if unproved, it would not block the `SOUND` verdict.
- **Struct invariants** are checked at every return point and at loop-back edges inside `drop`, ensuring invariants hold at all observable states.

## 8.5.3 Interpreting Results

| Result | Meaning | Action |
|--------|---------|--------|
| All `Proved` → `SOUND` | All contracts proved for all paths. | Function is verified. |
| Some `Unproved` → `UNSOUND` | At least one property could not be proved. | Inspect the unproved property and path. May indicate a missing annotation or genuine UB. |
| `Unknown` | Contract tag not yet supported. | Does not block soundness. |

## 8.5.4 Limitations

- **Path enumeration** is capped. Functions with very complex control flow may exceed the limit.
- **Inter-procedural analysis** is limited to pre-defined call summaries; full cross-function verification is not supported.
- **SMT encoding** approximates numeric constraints; non-linear arithmetic may yield `Unproved`.
- **Standard library contracts** are maintained manually; functions not covered are silently skipped.
- **Not all property kinds** have SMT checkers; unrecognized tags produce `Unknown`.
- **Concurrent code** and **inline assembly** are not analyzed.
